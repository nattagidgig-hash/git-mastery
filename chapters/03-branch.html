Branch

branch คือ pointer เบา ๆ ที่ชี้ไปยัง commit หนึ่ง สร้างได้ในพริบตา ทำให้คุณแตกสายงานไปลองของใหม่ได้โดยไม่กระทบสายหลัก เมื่อเข้าใจว่า branch กับ HEAD เป็นแค่ pointer คำสั่งทุกตัวจะชัดเจนทันที

🎯 Learning Objectives
  • เข้าใจว่า branch คือ movable pointer ไปยัง commit (ไม่ใช่สำเนาของไฟล์)
  • สร้าง/ลิสต์/สลับ branch ด้วย git branch และ git switch
  • เทียบ switch สมัยใหม่กับ checkout แบบเดิม และรู้ว่า HEAD ขยับเมื่อใด
  • เข้าใจ detached HEAD ว่าเกิดขึ้นเมื่อไรและอันตรายตรงไหน

1. branch คืออะไรกันแน่

หลายคนคิดว่า branch คือ "การก๊อปปี้โฟลเดอร์ทั้งหมดไปอีกชุด" — ไม่ใช่เลย ใน Git branch คือไฟล์เล็ก ๆ ที่เก็บ hash ของ commit หนึ่งตัว (40 ตัวอักษร) แค่นั้น เวลาคุณสร้าง branch ใหม่ Git แค่เขียน pointer อีกอันชี้ไปยัง commit ปัจจุบัน จึงเร็วและกินพื้นที่แทบเป็นศูนย์

เมื่อคุณ commit บน branch ใด pointer ของ branch นั้น เดินหน้าไปยัง commit ใหม่ โดยอัตโนมัติ ส่วน branch อื่นยังอยู่ที่เดิม นี่คือหัวใจของการ "แตกสายงาน"

cat .git/refs/heads/main
a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0   # branch ก็แค่ไฟล์ที่เก็บ hash หนึ่งตัว
💡
Mental model: commit คือสายโซ่ของ snapshot ส่วน branch คือ "ป้ายชื่อ" ที่แปะอยู่บน commit หนึ่ง และเลื่อนตามทุกครั้งที่ commit ใหม่บนสายนั้น

2. แตกสายงาน: ภาพรวม

สมมติคุณอยู่บน main แล้วอยากลองทำฟีเจอร์ใหม่โดยไม่ให้สายหลักพัง คุณสร้าง branch feature แล้ว commit ต่อบนนั้น ผลคือสายงานแยกออกเป็นสองทาง ภาพด้านล่างกด ถัดไป เพื่อดูทีละขั้น

3. คำสั่งพื้นฐานของ branch

สามสิ่งที่ทำบ่อยที่สุดคือ ลิสต์ branch, สร้าง branch, และสลับไปยัง branch

git branch                  # ลิสต์ branch ทั้งหมด, * นำหน้าอันที่อยู่ตอนนี้
* main
  feature

git branch feature          # สร้าง branch ชื่อ feature (ยังไม่สลับไป)
git switch feature          # สลับไปยัง feature — HEAD ขยับไปที่นั่น
git switch -c feature       # สร้าง + สลับในคำสั่งเดียว (ทางลัดที่ใช้บ่อยสุด)
⚠️
git branch feature แค่ สร้าง pointer ใหม่ แต่ ไม่สลับไปให้ — คุณยังยืนอยู่ที่ branch เดิม ถ้า commit ต่อก็จะลงสายเดิม ระวังสับสนตรงนี้บ่อย

4. HEAD ขยับเมื่อ switch

HEAD คือ pointer ที่บอกว่า "ตอนนี้คุณยืนอยู่ที่ branch ไหน" ปกติ HEAD ไม่ได้ชี้ commit ตรง ๆ แต่ชี้ผ่าน branch อีกที (เรียกว่า symbolic ref) เมื่อคุณ git switch feature สิ่งที่เปลี่ยนคือ HEAD เลิกชี้ main หันไปชี้ feature แทน แล้ว Git ก็ปรับไฟล์ใน Working Directory ให้ตรงกับ commit ปลายทาง

ลำดับการชี้คือ HEAD → feature → e5 เมื่อคุณ commit ใหม่ pointer feature จะเลื่อนไปยัง commit ใหม่ และเพราะ HEAD ชี้ผ่าน feature มันจึงเลื่อนตามไปด้วย

5. switch (สมัยใหม่) vs checkout (เดิม)

เดิม Git มีคำสั่ง git checkout ตัวเดียวที่ทำหลายหน้าที่ปนกัน — ทั้งสลับ branch, สร้าง branch, และกู้ไฟล์ — ทำให้สับสนและเผลอทำพังได้ง่าย Git 2.23 (ปี 2019) จึงแยกออกเป็นสองคำสั่งที่ชัดเจน: git switch สำหรับ branch และ git restore สำหรับไฟล์

เป้าหมายแบบเดิม (checkout)สมัยใหม่ (switch)
สลับไปยัง branch ที่มีอยู่git checkout featuregit switch feature
สร้าง + สลับ branch ใหม่git checkout -b featuregit switch -c feature
กลับไป branch ก่อนหน้าgit checkout -git switch -
กู้ไฟล์ในโฟลเดอร์git checkout -- filegit restore file
✅
ทั้งสองยังใช้ได้ปกติ แต่สำหรับงานเกี่ยวกับ branch แนะนำ switch เพราะตั้งใจทำเรื่องเดียวคือสลับสาย ลดโอกาสเผลอไปแตะไฟล์ checkout ยังเหมาะกับงานขั้นสูงบางอย่าง (เช่น checkout commit hash ตรง ๆ)

6. Detached HEAD: เมื่อ HEAD ชี้ commit ตรง ๆ

ปกติ HEAD ชี้ผ่าน branch แต่ถ้าคุณ git checkout <commit-hash> หรือ git switch --detach <hash> HEAD จะ ชี้ที่ commit นั้นโดยตรง ไม่ผ่าน branch ใด ๆ สถานะนี้เรียก detached HEAD มีไว้เพื่อ "ดู" หรือทดลองบน commit เก่า

git checkout a1b2c3d         # ไปยืนบน commit เก่าโดยตรง
You are in 'detached HEAD' state. ...
🚨
อันตราย: ถ้าคุณ commit ในสถานะ detached HEAD แล้วสลับ branch ออกไปโดยไม่ตั้งชื่อ branch ให้ commit เหล่านั้น มันจะ "ลอย" ไม่มี pointer ชี้ และอาจถูกลบทิ้งตอน garbage collection ทางออก: ก่อนออก ให้ git switch -c <ชื่อใหม่> เพื่อปักหมุดงานไว้บน branch จริง

วิธีออกจาก detached HEAD แบบปลอดภัย: ถ้าไม่ได้ commit อะไรก็แค่ git switch main กลับมา ถ้า commit ไปแล้วและอยากเก็บ ให้สร้าง branch จากตรงนั้นก่อน

7. ลองเอง — Interactive Terminal

นี่คือ git CLI จำลอง ลองวงจรแตกสายงาน: เริ่มจาก main → commit → สร้างและสลับไป feature → commit แยกสาย → ดู log

⌨️
ลองพิมพ์ต่อ: git switch -c feature → echo "feature work" > app.txt → git add . → git commit -m "work on feature" แล้ว git branch และ git log --oneline สังเกตว่า feature เดินไปข้างหน้าส่วน main ยังอยู่ที่เดิม จากนั้น git switch main เพื่อกลับมา

9. Workshop

🛠️
ภารกิจ: เปิด terminal จริงในเครื่องคุณ ฝึกแตกสายงานและสังเกต pointer ทุกขั้น
  1. สร้างโฟลเดอร์ใหม่ → git init → สร้างไฟล์ index.html แล้ว git add . และ git commit -m "init"
  2. git branch ดูว่ามี branch อะไรบ้าง (ควรมีแค่ main)
  3. git switch -c feature/login → รัน git branch อีกครั้ง สังเกต * ย้ายมาที่ feature/login
  4. แก้ไฟล์ → git add . → git commit -m "add login" แล้ว git log --oneline --all ดูว่าสองสายแยกกัน
  5. git switch main แล้วเปิดไฟล์ดู — เนื้อหากลับเป็นเวอร์ชันก่อนแก้ (อธิบายว่าทำไม)
  6. ลอง git switch - เพื่อเด้งกลับไป feature/login โดยไม่ต้องพิมพ์ชื่อเต็ม
เป้าหมาย: เห็นว่า branch แต่ละอันถือ snapshot คนละจุด และไฟล์ในโฟลเดอร์เปลี่ยนตาม HEAD ที่ขยับ

สรุป

  • branch = movable pointer ไปยัง commit หนึ่ง เป็นแค่ไฟล์เก็บ hash จึงเบาและสร้างได้ทันที
  • git branch ลิสต์, git branch <name> สร้าง (แต่ไม่สลับ), git switch <name> สลับ, git switch -c สร้าง+สลับในครั้งเดียว
  • HEAD ขยับเมื่อ switch และ commit บน branch ทำให้ pointer ของ branch (พร้อม HEAD) เดินหน้า ส่วนสายอื่นอยู่ที่เดิม
  • switch/restore สมัยใหม่แยกหน้าที่ชัดกว่า checkout แบบเดิมที่ทำหลายอย่างปนกัน
  • detached HEAD คือ HEAD ชี้ commit ตรง ๆ ไม่ผ่าน branch — ระวัง commit หาย ให้ git switch -c ปักหมุดก่อนออก
