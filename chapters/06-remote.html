บทที่ 6

Remote

repo บนเครื่องคุณกับ repo บน GitHub คือ repo คนละตัวที่คุยกันผ่าน "remote" เข้าใจว่า fetch ต่างจาก pull ยังไง แล้วคุณจะเลิกกลัวคำว่า "sync ไม่ตรง" ตลอดไป

🎯 Learning Objectives
  • เข้าใจว่า remote และ origin คืออะไร อ่าน git remote -v ได้
  • แยกชัด ๆ ระหว่าง git fetch (ดึงมาเฉย ๆ) กับ git pull (ดึง + merge)
  • เข้าใจ remote-tracking branch (origin/main) ว่าเป็นสำเนา read-only ของฝั่ง remote
  • ตั้ง upstream / tracking branch ด้วย git push -u และ --set-upstream-to

1. remote คืออะไร และ origin มาจากไหน

"remote" คือ ชื่อย่อ (alias) ที่ชี้ไปยัง URL ของ repo อีกตัวหนึ่ง โดยทั่วไปคือ repo บน GitHub/GitLab แทนที่จะต้องพิมพ์ URL ยาว ๆ ทุกครั้ง Git ให้เราตั้งชื่อสั้น ๆ แทน ชื่อ origin เป็นเพียง ชื่อตามธรรมเนียม ที่ git clone ตั้งให้อัตโนมัติ ไม่ใช่คำสงวน — คุณจะตั้งชื่ออื่นก็ได้

git remote -v          # ดู remote ทั้งหมด พร้อม URL (verbose)
origin  https://github.com/you/project.git (fetch)
origin  https://github.com/you/project.git (push)

สังเกตว่ามีสองบรรทัด — Git แยก URL สำหรับ fetch (ดึงลง) กับ push (ส่งขึ้น) ได้ แม้ส่วนใหญ่จะเป็น URL เดียวกัน คำสั่งจัดการ remote อื่น ๆ:

git remote add origin <url>   # เพิ่ม remote ชื่อ origin
git remote rename origin upstream  # เปลี่ยนชื่อ
git remote remove origin           # ลบออก
git remote show origin             # ดูรายละเอียด branch ทั้งหมดของ remote
💡
git remote และเพื่อน ๆ (fetch, pull, push) คุยกับเน็ตเวิร์ก — terminal จำลองในไซต์นี้ไม่รองรับ ตัวอย่างในบทนี้จึงเป็น <pre> ให้อ่าน แล้วลองจริงในเครื่องคุณ

2. repo สามตัว: local, remote-tracking, remote จริง

หัวใจของบทนี้คือต้องเห็นภาพว่ามี branch อยู่ สามชั้น ไม่ใช่สองชั้น หลายคนพลาดเพราะนึกว่ามีแค่ "เครื่องเรา" กับ "GitHub"

  • local branch (main) — branch ที่คุณ commit ลงไปได้จริง อยู่บนเครื่องคุณ
  • remote-tracking branch (origin/main) — สำเนา read-only ที่ Git จดไว้ว่า "ครั้งล่าสุดที่คุยกับ server main ของมันอยู่ตรงไหน" คุณ commit ลงตรงนี้ตรง ๆ ไม่ได้ มันขยับเมื่อคุณ fetch/pull/push เท่านั้น
  • remote จริง — main ตัวจริงที่อยู่บน GitHub
✅
origin/main ไม่ใช่ "ของบน GitHub แบบสด ๆ" แต่เป็น ภาพถ่ายครั้งล่าสุด ที่เครื่องคุณรู้ ถ้าเพื่อน push ขึ้นไปเมื่อ 5 นาทีก่อน เครื่องคุณจะยัง ไม่รู้ จนกว่าจะ fetch

3. git fetch — ดึงมาเก็บ แต่ไม่แตะงานคุณ

git fetch ไปถาม server ว่า "มี commit อะไรใหม่ไหม" แล้ว ดาวน์โหลด commit ใหม่ลงมาอัปเดตเฉพาะ origin/main เท่านั้น มัน ไม่แตะ working directory และไม่แตะ local branch main ของคุณเลย งานที่คุณกำลังทำอยู่จึงปลอดภัย 100% — fetch เป็นคำสั่งที่ "ปลอดภัยเสมอ" รันได้บ่อยเท่าที่ต้องการ

git fetch origin       # ดึง commit ใหม่มาที่ origin/main เท่านั้น
From github.com:you/project
   3a1f9c2..7d4e8b1  main -> origin/main

git log --oneline main..origin/main   # ดูว่ามี commit อะไรที่ตามหลังอยู่
7d4e8b1 add login form

หลัง fetch แล้ว local main ของคุณยังอยู่ที่เดิม คุณจะตรวจดูของใหม่ก่อน (git diff main origin/main) แล้วค่อยตัดสินใจ merge เมื่อไรก็ได้ นี่คือข้อดีของ fetch: แยกขั้น "ดึง" ออกจากขั้น "รวม"

4. git pull — fetch แล้ว merge ให้เลยในคำสั่งเดียว

git pull ไม่ใช่คำสั่งวิเศษ แต่เป็น ทางลัดของสองคำสั่ง:

git pull origin main
# เทียบเท่ากับ:
git fetch origin            # 1) ดึง commit ใหม่ → อัปเดต origin/main
git merge origin/main       # 2) merge เข้า branch ปัจจุบันทันที

เพราะ pull ทำขั้น merge ให้ทันที มันจึง เปลี่ยน working directory และ local branch ของคุณ — และถ้าของใหม่ชนกับงานที่คุณยังไม่ commit ก็อาจเกิด merge conflict ได้ (ดูบทที่ 7) ด้วยเหตุนี้ pull จึงไม่ใช่คำสั่งที่ "ปลอดภัยเสมอ" แบบ fetch

⚠️
commit งานคุณก่อน pull เสมอ ถ้ามีไฟล์แก้ค้างอยู่ใน working directory แล้ว pull ทับ อาจปวดหัวเรื่อง conflict หรือถูกขัดจังหวะกลางคัน นิสัยที่ดี: git status ให้ clean → แล้วค่อย pull
🧭
อยากให้ pull รวมแบบเส้นตรง (ไม่สร้าง merge commit) ใช้ git pull --rebase ซึ่งจะเอา commit ของคุณไปต่อท้ายของใหม่แทน (รายละเอียด rebase อยู่บทที่ 10)

6. push — ส่งงานคุณขึ้น server

git push ทำตรงข้ามกับ fetch: เอา commit ของ local branch ส่งขึ้นไปอัปเดต branch บน server โดยตรง (และอัปเดต origin/main ในเครื่องคุณตามไปด้วย) เงื่อนไขสำคัญ: push จะสำเร็จก็ต่อเมื่อ local ของคุณ นำหน้า (ahead) ของ server เท่านั้น ถ้า server มี commit ที่คุณยังไม่มี Git จะปฏิเสธ:

git push origin main
! [rejected]        main -> main (fetch first)
error: failed to push some refs ...
# แก้: git pull (รวมของใหม่เข้ามาก่อน) แล้วค่อย push ใหม่
🚫
เมื่อโดน rejected อย่ารีบ git push --force เพราะมันจะเขียนทับประวัติของเพื่อนที่ push ไปก่อน ทางที่ถูกคือ pull ให้รวมของใหม่เข้ามาก่อน ถ้าจำเป็นต้อง force จริง ๆ ให้ใช้ --force-with-lease ซึ่งปลอดภัยกว่า (จะ fail ถ้ามีคนอื่น push แทรกเข้ามา)

7. upstream / tracking branch

"tracking branch" คือการ จับคู่ local branch กับ remote branch หนึ่งตัว เมื่อจับคู่แล้ว คุณพิมพ์แค่ git push หรือ git pull เปล่า ๆ ได้เลย ไม่ต้องระบุ origin main ทุกครั้ง เพราะ Git จำได้ว่า local main ผูกกับ (มี upstream เป็น) origin/main

วิธีตั้ง upstream มีสองทาง:

# ทาง 1: ตอน push ครั้งแรก ใส่ -u (ย่อจาก --set-upstream)
git push -u origin main
Branch 'main' set up to track 'origin/main'.

# ทาง 2: ตั้งให้ branch ที่มีอยู่แล้ว
git branch --set-upstream-to=origin/main main

หลังตั้งแล้ว git status จะบอกได้ว่าคุณ ahead/behind server กี่ commit และคำสั่งสั้นลง:

git status -sb
## main...origin/main [ahead 2]   # มี 2 commit ที่ยังไม่ push

git push              # ไม่ต้องพิมพ์ origin main แล้ว
git branch -vv          # ดูว่าแต่ละ branch track อะไรอยู่
* main 7d4e8b1 [origin/main: ahead 2] add cart

9. Workshop

🛠️
ภารกิจ: ฝึก fetch/pull/push ให้เห็นความต่างจริง (ต้องมี repo บน GitHub สักตัว หรือสร้างใหม่ก็ได้) สังเกต git status -sb ทุกขั้น
  1. โคลน repo ลงมา: git clone <url> แล้วรัน git remote -v ดูว่า origin ชี้ไปไหน
  2. ไปแก้ไฟล์ผ่านหน้าเว็บ GitHub โดยตรง 1 commit (จำลองว่า "เพื่อน push") — ตอนนี้ server นำหน้าเครื่องคุณแล้ว
  3. กลับมาที่เครื่อง รัน git fetch → สังเกตว่า git log --oneline main ยังไม่เปลี่ยน แต่ git log --oneline origin/main มี commit ใหม่โผล่มา
  4. เทียบของใหม่กับของคุณ: git diff main origin/main
  5. รวมของใหม่เข้ามา: git merge origin/main (หรือทำขั้น 3-5 รวบเดียวด้วย git pull) — ตอนนี้ main ขยับตามแล้ว
  6. commit งานของคุณเอง 1 อัน แล้ว git push -u origin main → จากนั้นลอง git status -sb ดูว่าขึ้นว่า track origin/main และ ahead/behind เท่าไร
เป้าหมาย: เห็นกับตาว่า fetch ไม่แตะ main แต่ pull แตะ และเข้าใจว่า origin/main ขยับเมื่อไร

สรุป

  • remote คือชื่อย่อที่ชี้ไป URL ของ repo อีกตัว — origin เป็นเพียงชื่อตามธรรมเนียมที่ clone ตั้งให้ ดูได้ด้วย git remote -v
  • origin/main คือ remote-tracking branch — สำเนา read-only ของสถานะ remote ครั้งล่าสุดที่เครื่องคุณรู้
  • fetch ดึง commit ใหม่มาอัปเดต origin/main เท่านั้น ไม่แตะ local/working — ปลอดภัยเสมอ
  • pull = fetch + merge เข้า branch ปัจจุบันทันที จึงเปลี่ยนงานคุณและอาจเกิด conflict
  • ตั้ง upstream ด้วย git push -u หรือ --set-upstream-to แล้ว push/pull เปล่า ๆ ได้เลย
