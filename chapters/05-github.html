Git Mastery
บทที่ 5

GitHub

จนถึงตอนนี้ repo ของคุณอยู่แค่ในเครื่องตัวเอง GitHub คือที่ที่ทำให้มัน "ขึ้นคลาวด์" — สำรองไว้ที่ปลอดภัย แชร์ให้คนอื่น และทำงานร่วมกันด้วย Pull Request ได้

🎯 Learning Objectives
  • เข้าใจความสัมพันธ์ local ↔ remote (origin) และบทบาทของ GitHub
  • ทำขั้นตอน push โปรเจกต์ขึ้น GitHub ครั้งแรกได้: remote add → branch -M → push -u
  • แยกความต่างของ clone กับ fork และรู้ว่าเมื่อไหร่ใช้อันไหน
  • อธิบายวงจร Pull Request: เสนอ merge + code review + discussion

1. remote คืออะไร และ GitHub อยู่ตรงไหน

repo ในเครื่องคุณ (local) สมบูรณ์ในตัวเอง — มีประวัติ commit ครบ ทำงานออฟไลน์ได้หมด แต่ถ้าโน้ตบุ๊กพังหรืออยากให้เพื่อนช่วยทำ คุณต้องมีสำเนาที่ "อยู่ตรงกลาง" ที่ทุกคนเข้าถึงได้ นั่นคือ remote — repo อีกชุดที่อยู่บนเซิร์ฟเวอร์

GitHub คือบริการโฮสต์ remote repo บนคลาวด์ พ่วงเครื่องมือทำงานร่วมกัน (Pull Request, code review, issue, CI) มาให้ ชื่อ remote เริ่มต้นที่ทุกคนใช้กันคือ origin — เป็นแค่ชื่อเล่น(alias) ที่ชี้ไปยัง URL ของ repo บน GitHub

💡
origin เป็นแค่ชื่อ: ไม่ใช่คำสงวนของ Git ตั้งชื่อ remote เป็นอะไรก็ได้ แต่ origin คือธรรมเนียมมาตรฐานสำหรับ remote หลัก ทุกคนจึงเข้าใจตรงกัน

2. push โปรเจกต์ขึ้น GitHub ครั้งแรก

สมมติคุณมี repo ในเครื่อง (commit ไปบ้างแล้ว) และเพิ่งสร้าง repo เปล่าบน GitHub ขั้นตอนเชื่อมทั้งสองเข้าด้วยกันมี 4 บรรทัด ทำครั้งเดียวต่อโปรเจกต์

# 1) บอก local ว่า remote ชื่อ origin อยู่ที่ URL นี้
git remote add origin https://github.com/your-name/your-repo.git

# 2) ตั้งชื่อ branch ปัจจุบันให้เป็น main (มาตรฐานของ GitHub)
git branch -M main

# 3) push ครั้งแรก: -u ผูก local main เข้ากับ origin/main
git push -u origin main
Enumerating objects: 6, done.
To https://github.com/your-name/your-repo.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.

หลังจากนั้น เพราะ -u (ย่อจาก --set-upstream) ผูก local main ไว้กับ origin/main แล้ว ครั้งต่อ ๆ ไปคุณพิมพ์สั้นได้เลย:

git add .
git commit -m "add feature"
git push          # ไม่ต้องระบุ origin main อีก Git จำให้
⚠️
git push ไม่ใช่ git commit: commit บันทึกลง local เท่านั้น งานของคุณยัง ไม่ อยู่บน GitHub จนกว่าจะ push ถ้าลืม push เพื่อนจะไม่เห็นงานคุณ และไม่มี backup บนคลาวด์
🔒
HTTPS URL จะให้ใส่ Personal Access Token แทนรหัสผ่าน (GitHub ปิดการ auth ด้วย password แล้ว) หรือใช้ SSH URL (git@github.com:...) กับ SSH key ก็ได้ — สะดวกกว่าเพราะไม่ต้องกรอกทุกครั้ง

ภาพรวมของ commit ที่ local กับ remote หลัง push ครั้งแรก: ทั้งสองฝั่งชี้ที่ commit เดียวกัน

3. clone: ดึงทั้ง repo ลงมา

ถ้า repo อยู่บน GitHub อยู่แล้ว (ของคุณเองหรือของคนอื่นที่เปิดสาธารณะ) คุณไม่ต้อง init เอง แค่ clone ก็ได้ทั้งประวัติ commit + ตั้ง origin ให้อัตโนมัติ + checkout branch หลักให้พร้อมทำงาน

git clone https://github.com/some-org/cool-project.git
Cloning into 'cool-project'...
remote: Enumerating objects: 240, done.
Receiving objects: 100% (240/240), done.

cd cool-project
git remote -v          # origin ถูกตั้งให้แล้วโดยอัตโนมัติ
origin  https://github.com/some-org/cool-project.git (fetch)
origin  https://github.com/some-org/cool-project.git (push)

4. fork: สำเนา repo ภายใต้บัญชีเรา

ถ้าอยากแก้ repo ของ คนอื่น ที่คุณไม่มีสิทธิ์ push คุณ fork ก่อน — กดปุ่มบนหน้าเว็บ GitHub เพื่อสร้างสำเนา repo นั้นไว้ใต้บัญชีคุณเอง จากนั้นค่อย clone สำเนานั้นลงมาทำงาน แก้เสร็จก็เปิด Pull Request กลับไปยังต้นฉบับ

git clonefork
ทำที่ไหนคำสั่งในเครื่อง (CLI)ปุ่มบนหน้าเว็บ GitHub
สำเนาไปอยู่ไหนลงเครื่อง local ของคุณเป็น repo ใหม่ใต้บัญชี GitHub ของคุณ
สิทธิ์ pushต้องมีสิทธิ์ใน repo ต้นทางอยู่แล้วคุณเป็นเจ้าของสำเนา push ได้เต็มที่
ใช้เมื่อrepo ของตัวเอง/ทีมที่มีสิทธิ์อยากร่วม contribute repo สาธารณะของคนอื่น
✅
วงจร open-source ทั่วไป: fork → clone สำเนา → สร้าง branch → commit → push ขึ้น fork ของตัวเอง → เปิด Pull Request ไปยัง repo ต้นฉบับ

5. Pull Request: เสนอ merge + review + discussion

Pull Request (PR) ไม่ใช่คำสั่ง Git แต่เป็นฟีเจอร์ของ GitHub มันคือการ "ยื่นเรื่องขอ merge" branch ของคุณเข้า branch หลัก พร้อมพื้นที่ให้ทีมรีวิวโค้ด คอมเมนต์ทีละบรรทัด ถกเถียง และให้ CI รันทดสอบ ก่อนกดรวมจริง — เป็นหัวใจของการทำงานร่วมกันบน GitHub

# flow ฝั่ง local ก่อนเปิด PR
git switch -c feature/login
# ...แก้โค้ด แล้ว add/commit...
git push -u origin feature/login
remote: Create a pull request for 'feature/login' on GitHub by visiting:
remote:   https://github.com/you/repo/pull/new/feature/login

คลิกลิงก์นั้นจะพาไปหน้าสร้าง PR บน GitHub กรอกคำอธิบาย เลือก reviewer แล้วเปิด PR ทีมรีวิว เห็นชอบ → กด Merge งานของคุณก็เข้า main วงจรของ PR:

📝
PR ที่ดีเล่าเรื่อง: เขียนว่าแก้อะไร ทำไม ทดสอบยังไง — รีวิวเวอร์ (และตัวคุณเองในอีก 6 เดือน) จะขอบคุณ commit message สั้นกระชับ แต่ PR description คือที่อธิบาย "ภาพใหญ่"

6. README: หน้าตาแรกของ repo

ไฟล์ README.md ที่ root ของ repo จะถูก GitHub เรนเดอร์เป็นหน้าแรกอัตโนมัติ มันคือ "หน้าร้าน" ของโปรเจกต์ — บอกว่าโปรเจกต์คืออะไร ติดตั้ง/รันยังไง contribute ยังไง เขียนด้วย Markdown

# README.md (Markdown)
# Cool Project
เครื่องมือช่วย ... อธิบายสั้น ๆ ว่าโปรเจกต์ทำอะไร

## Install
    npm install

## Usage
    npm start

8. Workshop

🛠️
ภารกิจ: push โปรเจกต์จริงของคุณขึ้น GitHub ครั้งแรก
  1. มี repo ในเครื่องที่ commit แล้วอย่างน้อย 1 ครั้ง (ถ้ายังไม่มี: git init → สร้างไฟล์ → git add . → git commit -m "init")
  2. เข้า github.com กดปุ่ม New repository สร้าง repo เปล่า (ยังไม่ต้องติ๊ก Add README เพื่อเลี่ยงประวัติชนกัน) แล้วคัดลอก URL
  3. ในเครื่อง: git remote add origin <url> แล้วเช็กด้วย git remote -v
  4. git branch -M main เพื่อให้ชื่อ branch ตรงกับ GitHub
  5. git push -u origin main — ใส่ token/credential แล้วรีเฟรชหน้า GitHub จะเห็นไฟล์ขึ้นมา
  6. เพิ่มไฟล์ README.md → git add → git commit → git push (คราวนี้สั้นได้) แล้วดูว่า GitHub เรนเดอร์ README เป็นหน้าแรก
เป้าหมาย: เห็นวงจร local → push → remote ครบ และเข้าใจว่าทำไมครั้งที่สอง push สั้นกว่าครั้งแรก

สรุป

  • remote คือ repo อีกชุดบนเซิร์ฟเวอร์ — GitHub โฮสต์ remote บนคลาวด์พร้อมเครื่องมือ collaboration
  • เชื่อมครั้งแรก: git remote add origin <url> → git branch -M main → git push -u origin main หลังจากนั้นแค่ git push
  • clone = ดึง repo ทั้งชุดลงเครื่อง (ตั้ง origin ให้เอง); fork = สำเนา repo คนอื่นมาไว้ใต้บัญชีเราเพื่อ contribute
  • Pull Request = เสนอ merge + code review + discussion ก่อนรวมเข้า main; README.md คือหน้าแรกของ repo
