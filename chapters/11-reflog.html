Reflog

เผลอ reset --hard, rebase พัง, หรือลบ branch ผิด แล้วงานหายเป็นสิบ commit? อย่าเพิ่งตกใจ — Git จดทุกก้าวของ HEAD ไว้ใน reflog เกือบทุกอย่าง "กู้ได้" ถ้ารู้จักเครื่องมือนี้

🎯 Learning Objectives
  • เข้าใจว่า reflog คือบันทึกประวัติทุกตำแหน่งที่ HEAD เคยอยู่
  • อ่านรูปแบบ HEAD@{n} และหา hash ของ commit ที่ "หาย" ได้
  • กู้คืนงานด้วย git reset --hard HEAD@{n} หรือ git checkout <hash> + สร้าง branch ใหม่
  • รู้ขอบเขตของ reflog: local เท่านั้น และ entry หมดอายุ (~90 วัน)

1. reflog คืออะไร

ทุกครั้งที่ HEAD ขยับ — ไม่ว่าจะ commit, switch/checkout, reset, rebase, หรือ merge — Git จะจดบรรทัดหนึ่งลงใน reference log (reflog) ว่า "HEAD ย้ายจากไหนไปไหน เพราะคำสั่งอะไร" reflog จึงเป็นเหมือน กล้องวงจรปิด ของ repo คุณ: แม้ commit จะหลุดออกจากทุก branch จนมองไม่เห็นใน git log ตัว commit object ก็ยังอยู่ใน .git และ reflog ยังเก็บ hash ของมันไว้ให้คุณเดินกลับไปหาได้

💡
กุญแจสำคัญ: git log เดินตามสายโซ่ parent ของ branch ปัจจุบัน ส่วน git reflog เดินตาม ลำดับเวลา ที่ HEAD เคยขยับ — มันเห็น commit ที่ git log มองไม่เห็นแล้ว

2. รูปแบบ HEAD@{n}

reflog ใช้พิกัดแบบ HEAD@{n} โดย n นับ ถอยหลังจากปัจจุบัน: HEAD@{0} คือตำแหน่งล่าสุด, HEAD@{1} คือก่อนหน้านั้นหนึ่งก้าว ไปเรื่อย ๆ ลองรัน git reflog ดูผลแบบนี้:

git reflog
e3a9f12 HEAD@{0}: reset: moving to HEAD~2
a7c4d90 HEAD@{1}: commit: add payment validation
b1f8e23 HEAD@{2}: commit: add cart total
e3a9f12 HEAD@{3}: commit: setup checkout page
9d2c0a8 HEAD@{4}: checkout: moving from main to feature

แต่ละบรรทัดบอกครบ: hash ปลายทาง, พิกัด HEAD@{n}, ชนิดของการกระทำ (commit, reset, checkout...), และข้อความอธิบาย จากตัวอย่างจะเห็นว่า HEAD@{0} เป็นผลของ reset ที่ย้อนกลับไปยัง commit เก่า — แปลว่า commit a7c4d90 ("add payment validation") ตอนนี้ไม่อยู่บน branch แล้ว แต่ hash ยังอยู่ตรงนี้

📝
นอกจาก HEAD@{n} แล้ว reference อื่นก็มี reflog ของตัวเองด้วย เช่น git reflog show main ดูเฉพาะการขยับของ branch main

3. สถานการณ์: reset --hard ทำ commit หาย

มาดูภาพว่า commit "หาย" ได้ยังไง สมมติคุณอยู่ที่ c3 แล้วเผลอรัน git reset --hard HEAD~2 เพื่อย้อนไป a1 — branch main ขยับกลับ แต่ b2 และ c3 ยังลอยอยู่ใน .git โดยไม่มี branch ชี้ (เรียก dangling commits)

ตอนนี้ git log เห็นแค่ a1 เหมือนงานหายไปหมด — แต่ git reflog จะยังโชว์ hash ของ c3 ให้

⚠️
git reset --hard ทำสองอย่างพร้อมกัน: ย้าย branch และ ล้าง working directory ทิ้ง การเปลี่ยนแปลงที่ยัง ไม่เคย commit (uncommitted) จะหายจริง ๆ และ reflog ช่วยไม่ได้ — reflog กู้ได้เฉพาะสิ่งที่ "เคยเป็น commit" เท่านั้น

4. กู้คืน: สองวิธี

เมื่อหา hash จาก reflog เจอแล้ว มีสองทางหลักในการกู้:

วิธี A — reset --hard กลับไปที่จุดนั้น (ย้าย branch ปัจจุบันกลับ)

git reflog                         # หา HEAD@{n} ของ commit ที่ต้องการ
git reset --hard HEAD@{1}        # ดึง main กลับไปที่ c3 อีกครั้ง
git reset --hard a7c4d90        # หรือใช้ hash ตรง ๆ ก็ได้ ผลเหมือนกัน

เหมาะเมื่อคุณ แน่ใจ ว่าอยากให้ branch ปัจจุบันกลับไปอยู่จุดนั้นเป๊ะ ๆ

วิธี B — checkout hash แล้วสร้าง branch ใหม่ (ปลอดภัยกว่า)

git checkout a7c4d90              # ไปยืนที่ commit นั้น (detached HEAD)
git switch -c rescue-work        # ปักธง branch ใหม่ตรงนี้ ไม่แตะ main
# ตอนนี้ commit ที่หายถูก "ผูก" กับ branch rescue-work แล้ว ปลอดภัย
✅
ก่อนตกใจว่างานหาย ลอง git reflog ก่อนเสมอ ถ้าสิ่งที่หายเคยถูก commit แล้ว มันแทบไม่เคยหายจริง วิธี B ปลอดภัยที่สุด เพราะแค่ "ปักธงใหม่" โดยไม่เขียนทับ branch ปัจจุบัน — ตรวจดูให้พอใจก่อนค่อย merge กลับ

5. ตารางเปรียบเทียบ: log vs reflog

git loggit reflog
เดินตามอะไรสายโซ่ parent ของ branchลำดับเวลาที่ HEAD ขยับ
เห็น dangling commit ไหมไม่เห็นเห็น (ตราบใดที่ยังไม่หมดอายุ)
แชร์ขึ้น remote ไหมใช่ (เป็นส่วนของประวัติ)ไม่ — local เท่านั้น
อยู่ถาวรไหมถาวร (จนกว่าจะลบ)หมดอายุ ~90 วัน (default)
เหมาะกับดูประวัติโปรเจกต์กู้งานที่เพิ่งทำหาย
🚨
reflog เป็น local เท่านั้น — มันไม่ถูก push และไม่ตามมาเวลา clone ถ้าคุณ clone repo ใหม่ในเครื่องอื่น reflog จะว่างเปล่า อย่าพึ่งพา reflog เป็น backup ระยะยาว และอย่ารัน git reflog expire --expire=now --all ตามมั่ว — มันลบตาข่ายนิรภัยของคุณทิ้ง

6. reflog ครอบคลุมอะไรบ้าง

reflog ช่วยกู้ได้ในหลายสถานการณ์ที่ดูเหมือนหายนะ ตราบใดที่สิ่งนั้น เคยถูก commit:

กรณีลบ branch ผิด: ตอนลบ Git จะพิมพ์ hash ออกมาด้วย (เช่น Deleted branch feature (was a7c4d90)) เอา hash นั้นไป git checkout แล้วสร้าง branch ใหม่ได้เลย หรือถ้าจำไม่ได้ก็หาใน reflog ของ HEAD

8. Workshop

🛠️
ภารกิจ: ทำ commit หายแล้วกู้คืนด้วย reflog เปิด terminal จริงในเครื่องคุณ แล้วทำตามทีละขั้น
  1. สร้างโฟลเดอร์ใหม่ → git init → git commit --allow-empty -m "A"
  2. ทำซ้ำให้ได้สาม commit: "B" และ "C" แล้วรัน git log --oneline (ควรเห็น A, B, C)
  3. ทำพัง: git reset --hard HEAD~2 → รัน git log --oneline อีกครั้ง จะเหลือแค่ A เหมือน B,C หายไป 😱
  4. สืบหา: git reflog — มองหาบรรทัดที่เขียนว่า commit: C แล้วจดพิกัด HEAD@{n} หรือ hash ของมัน
  5. กู้คืน: ลองทั้งสองวิธี — (A) git reset --hard HEAD@{n} หรือ (B) git checkout <hash-ของC> แล้ว git switch -c rescue
  6. รัน git log --oneline ยืนยันว่า C กลับมาแล้ว
เป้าหมาย: เห็นกับตาว่า commit ที่ "หาย" หลัง reset --hard กู้กลับได้เสมอ ถ้ามันเคยถูก commit

สรุป

  • reflog = บันทึกทุกที่ที่ HEAD เคยอยู่ (commit, switch, reset, rebase, merge) — ตาข่ายนิรภัยของคุณ
  • ใช้พิกัด HEAD@{n} นับถอยหลัง: HEAD@{0} คือล่าสุด
  • กู้คืนด้วย git reset --hard HEAD@{n} (ย้าย branch กลับ) หรือ git checkout <hash> + สร้าง branch ใหม่ (ปลอดภัยกว่า)
  • reflog เป็น local เท่านั้น ไม่ push และ entry หมดอายุ ~90 วัน — ไม่ใช่ backup ระยะยาว
  • ก่อนตกใจว่างานหาย ลอง git reflog ก่อนเสมอ — สิ่งที่เคย commit แทบไม่เคยหายจริง
