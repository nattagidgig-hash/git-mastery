Worktree

เคยอยากเปิดสอง branch พร้อมกันแบบไม่ต้อง stash ทุกครั้งไหม? git worktree ให้คุณกาง branch หลายตัวออกมาเป็นโฟลเดอร์แยก ที่แชร์ .git เดียวกัน — แก้ hotfix ด่วนได้โดยไม่ทิ้งงาน feature ที่ค้างอยู่

🎯 Learning Objectives
  • เข้าใจว่า worktree คือ working directory เพิ่ม ที่แชร์ .git เดียวกัน
  • ใช้ git worktree add / list / remove ได้คล่อง
  • รู้ use case จริง: hotfix ด่วน, รัน test, review PR — โดยไม่ต้อง stash หรือ switch
  • เข้าใจ ข้อจำกัด: หนึ่ง branch checkout ได้ใน worktree เดียวเท่านั้น และต่างจาก clone

1. ปัญหา: งานค้าง แต่ต้องแก้ของด่วน

สถานการณ์คลาสสิก — คุณกำลังเขียน feature/checkout อยู่ครึ่งทาง ไฟล์ยังแก้ค้าง คอมไพล์ไม่ผ่าน แล้วจู่ ๆ มีบั๊ก production ต้อง hotfix บน main เดี๋ยวนี้ ทางเดิมคือ git stash → git switch main → แก้ → กลับมา git stash pop ซึ่งเสี่ยงลืม pop, conflict ตอน pop, และต้องหยุดงานเดิมทั้งหมด

worktree เสนอทางที่สะอาดกว่า: เปิด main เป็นอีกโฟลเดอร์ไปเลย งาน feature ของคุณยังอยู่ในโฟลเดอร์เดิมเป๊ะ ไม่ถูกแตะ ทำ hotfix ในโฟลเดอร์ใหม่เสร็จก็กลับมาทำงานต่อเหมือนไม่มีอะไรเกิดขึ้น

💡
Mental model: หนึ่ง repo มี .git หนึ่งก้อน (ฐานข้อมูลประวัติ) แต่สามารถมี หลาย working directory ที่ดึง branch คนละตัวออกมากางพร้อมกัน ทุกโฟลเดอร์เห็น commit/branch ชุดเดียวกันเพราะแชร์ .git

2. ภาพรวม: หนึ่ง .git หลาย working directory

ปกติ git clone ให้คุณ 1 working directory ผูกกับ 1 .git แต่ worktree ทำให้ .git ก้อนเดิมแตกแขนง working directory ออกไปได้หลายอัน แต่ละอัน checkout คนละ branch

3. คำสั่งหลัก: add, list, remove

worktree มีคำสั่งย่อยไม่กี่ตัว แต่ครอบคลุมงานทั้งหมด เริ่มจากการเพิ่ม:

git worktree add ../project-hotfix hotfix/login   # สร้างโฟลเดอร์ใหม่ + checkout branch ที่มีอยู่
Preparing worktree (checking out 'hotfix/login')
HEAD is now at a1b2c3d fix login redirect

ถ้าอยากสร้าง branch ใหม่พร้อมกับ worktree เลย ใช้ -b:

git worktree add -b hotfix/login ../project-hotfix main   # แตก branch ใหม่จาก main แล้วกางที่โฟลเดอร์ใหม่

ดูว่าตอนนี้มี worktree อะไรบ้างและชี้ commit ไหน:

git worktree list
/home/me/project           a1b2c3d [main]
/home/me/project-feature   9f8e7d6 [feature/checkout]
/home/me/project-hotfix    a1b2c3d [hotfix/login]

เมื่อใช้เสร็จ ลบ worktree ทิ้ง (ลบโฟลเดอร์ + ปลด register ออกจาก .git):

git worktree remove ../project-hotfix   # ลบ worktree ที่สะอาด (ไม่มีงานค้าง)
git worktree prune                      # เก็บกวาด record ของโฟลเดอร์ที่ถูกลบไปเองด้วยมือ
⚠️
ถ้าลบโฟลเดอร์ worktree ทิ้งด้วย rm -rf เฉย ๆ Git จะยังจำว่ามันมีอยู่ (เป็น stale entry) ให้รัน git worktree prune เพื่อล้าง record ค้าง หรือใช้ git worktree remove ตั้งแต่แรกจะสะอาดกว่า

4. เดินเรื่องด้วยกราฟ: hotfix ระหว่าง feature ค้าง

กราฟด้านล่างเล่าเรื่องจริง: feature/checkout กำลังดำเนินอยู่ จากนั้นเปิด worktree ใหม่บน hotfix/login แตกจาก main แก้เสร็จ merge กลับ main — ทั้งหมดนี้ feature/checkout ไม่ถูกแตะเลย กดปุ่มเพื่อไล่ทีละขั้น

5. worktree vs stash vs clone

สามวิธีนี้ตอบโจทย์ "ทำงานหลายอย่างพร้อมกัน" คนละมุม เลือกให้ถูกงาน:

เกณฑ์git worktreegit stashgit clone (สำเนาที่ 2)
เปิดหลาย branch พร้อมกัน✅ คนละโฟลเดอร์ เปิดพร้อมกันได้❌ ทีละ branch ในโฟลเดอร์เดียว✅ แต่เปลือง
แชร์ .git / ประวัติ✅ แชร์ก้อนเดียว✅ (อยู่ repo เดียว)❌ มี .git แยก ต้อง fetch เอง
พื้นที่ดิสก์เพิ่มเฉพาะ working filesแทบไม่เพิ่มเพิ่มทั้ง history + files (เปลืองสุด)
ต้องหยุดงานเดิม❌ งานเดิมคงอยู่✅ ต้อง stash ก่อน❌ งานเดิมคงอยู่
เหมาะกับhotfix ด่วน / รัน test / review PR ขนานกันพักงานสั้น ๆ ชั่วคราวแยก environment / สิทธิ์ remote ต่างกัน
✅
กฎหยาบ ๆ: พักงานแป๊บเดียวแล้วกลับมาเลย → stash | ต้องทำสองงานสลับไปมาเป็นชั่วโมง ๆ → worktree | ต้องการ .git คนละก้อนจริง ๆ → clone

6. ข้อจำกัดที่ต้องรู้

worktree ทรงพลังแต่มีกติกาเฉพาะตัว ที่สำคัญที่สุด:

  • หนึ่ง branch checkout ได้ใน worktree เดียวเท่านั้น — ถ้า main ถูกกางอยู่ที่โฟลเดอร์หลักแล้ว คุณจะ worktree add โฟลเดอร์อื่นด้วย main ซ้ำไม่ได้ Git จะเตือนทันที (กันสองโฟลเดอร์แก้ branch เดียวกันชนกัน)
  • ลบโฟลเดอร์ทิ้งเองต้อง prune ตามเสมอ ไม่งั้นมี stale entry
  • ไฟล์ที่ไม่อยู่ใน Git (เช่น node_modules, .env, build cache) ไม่ถูกแชร์ ระหว่าง worktree — แต่ละโฟลเดอร์ต้อง install/config เอง
git worktree add ../another-main main
fatal: 'main' is already used by worktree at '/home/me/project'
🚫
อย่าเปิด IDE/editor ตัวเดียวกันคร่อมหลาย worktree แล้วคาดหวังว่ามันจะเข้าใจว่าเป็น branch ต่างกัน — เปิดแต่ละโฟลเดอร์เป็น project แยก จะลดความสับสนและ build cache ชนกัน

8. Workshop

🛠️
ภารกิจ: เพิ่ม worktree สำหรับ hotfix ด่วน ขณะที่งาน feature ยังค้างอยู่ ทำในเครื่องจริงของคุณ (worktree ไม่รองรับใน simulator)
  1. ใน repo ที่มี commit อยู่แล้ว สร้าง branch งาน: git switch -c feature/x แล้วแก้ไฟล์สักไฟล์ แต่ยังไม่ commit (จำลองงานค้าง)
  2. รัน git worktree add -b hotfix/y ../proj-hotfix main — สังเกตว่าโฟลเดอร์ ../proj-hotfix ถูกสร้างและอยู่บน branch hotfix/y
  3. cd ../proj-hotfix แล้ว git status — ยืนยันว่างานค้างของ feature/x ไม่ ตามมาที่นี่ (โฟลเดอร์สะอาด)
  4. แก้บั๊กในโฟลเดอร์นี้ → git add . → git commit -m "hotfix y"
  5. กลับโฟลเดอร์เดิม git worktree list ดูทั้งสอง worktree แล้ว git worktree remove ../proj-hotfix เก็บกวาด
เป้าหมาย: เห็นว่างาน feature ที่ค้างอยู่ ไม่ถูกแตะ ตลอดทั้งกระบวนการ hotfix — ไม่ต้อง stash แม้แต่ครั้งเดียว

สรุป

  • git worktree add <path> <branch> สร้าง working directory เพิ่ม ที่แชร์ .git เดียวกัน เปิดหลาย branch พร้อมกันได้โดยไม่ต้อง stash/switch
  • git worktree list ดูทั้งหมด, git worktree remove <path> ลบ, git worktree prune ล้าง record ค้าง
  • Use case จริง: hotfix ด่วน ขณะ feature ค้าง, รัน test branch หนึ่งระหว่างเขียนอีก branch, review PR ขนานกัน
  • ข้อจำกัด: หนึ่ง branch checkout ได้ใน worktree เดียว และไฟล์นอก Git (node_modules/.env) ไม่ถูกแชร์
  • เทียบ: stash = พักสั้น ๆ, worktree = สองงานขนานในโฟลเดอร์แยกแต่ .git เดียว, clone = .git คนละก้อน
