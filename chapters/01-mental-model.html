Git Mental Model

ก่อนจะจำคำสั่ง ต้องเห็นภาพก่อนว่า Git เก็บข้อมูลยังไง เมื่อ "ภาพในหัว" ถูก คำสั่งทุกตัวจะมีเหตุผล ไม่ต้องท่อง

🎯 Learning Objectives
  • อธิบาย สามพื้นที่ ของ Git: Working Directory, Staging Area, Repository
  • เข้าใจว่า Git เก็บเป็น snapshot ไม่ใช่ความต่าง (diff)
  • รู้ว่า commit คืออะไร, hash มาจากไหน, และ HEAD ชี้อะไร
  • ลองวงจร add → commit ด้วยตัวเองใน terminal simulator

1. Git ไม่ได้เก็บ "ความต่าง" แต่เก็บ "ภาพถ่าย"

หลายคนเข้าใจผิดว่า Git เก็บรายการการเปลี่ยนแปลงทีละบรรทัด ความจริงคือ ทุกครั้งที่ commit Git ถ่าย snapshot ของไฟล์ทั้งโปรเจกต์ ณ ตอนนั้น ถ้าไฟล์ไหนไม่เปลี่ยน Git ก็แค่ชี้ไปยังเวอร์ชันเดิม (ไม่เก็บซ้ำ) นี่คือเหตุผลที่ Git เร็วและสลับเวอร์ชันได้ทันที

💡
Mental model หลัก: คิดว่าแต่ละ commit คือ "ภาพถ่ายของทั้งโฟลเดอร์" ที่ผูกกับภาพก่อนหน้าเป็นสายโซ่

2. สามพื้นที่ (The Three Trees)

งานของคุณเดินทางผ่านสามพื้นที่เสมอ เข้าใจสามอันนี้ = เข้าใจ Git ไป 80%

พื้นที่คืออะไรคำสั่งที่เกี่ยวข้อง
Working Directoryโฟลเดอร์จริงที่คุณเปิดแก้ไฟล์git status
Staging Areaห้องรอ — เลือกว่า commit หน้าจะมีอะไรบ้างgit add
Repository (.git)ฐานข้อมูลประวัติทั้งหมด commit แล้วอยู่ถาวรgit commit
✅
Staging area คือสิ่งที่ทำให้ Git ต่างจาก backup ทั่วไป — คุณเลือก commit เฉพาะส่วนที่ตั้งใจได้ ทำให้ประวัติเล่าเรื่องเป็นขั้นเป็นตอน

3. ข้างใน .git: object และ hash

ทุก commit ถูกระบุด้วย SHA-1 hash (เช่น a1b2c3d) ที่คำนวณจากเนื้อหา snapshot + ข้อมูล commit ก่อนหน้า เปลี่ยนอะไรแม้นิดเดียว hash ก็เปลี่ยน นี่คือเหตุผลที่ประวัติ Git ปลอมแปลงยาก แต่ละ commit ยังเก็บ pointer ไปยัง parent จึงร้อยกันเป็นสายโซ่

4. HEAD: "คุณอยู่ตรงไหน"

HEAD คือ pointer พิเศษที่บอกว่าตอนนี้คุณกำลังยืนอยู่ที่ commit ไหน (ปกติชี้ผ่าน branch เช่น main) เวลา commit ใหม่ HEAD จะขยับตามไปข้างหน้าโดยอัตโนมัติ เวลาสลับ branch ก็แค่ขยับ HEAD ไปชี้อีกสาย — นี่คือเหตุผลที่ branch ใน Git ถูกและเร็วมาก (เป็นแค่ pointer 40 ตัวอักษร)

cat .git/HEAD
ref: refs/heads/main      # HEAD ชี้ branch main

5. ลองเอง — Interactive Terminal

นี่คือ git CLI จำลอง สถานะเปลี่ยนจริงตามคำสั่ง ลองวงจรพื้นฐาน: สร้างไฟล์ → add → commit → ดู log

⌨️
ลองพิมพ์ต่อ: git add . แล้ว git commit -m "first commit" แล้ว git log --oneline — สังเกตว่า git status เปลี่ยนไปยังไงในแต่ละขั้น

7. Workshop

🛠️
ภารกิจ: เปิด terminal จริงในเครื่องคุณ แล้วทำตามนี้ พร้อมสังเกต git status ทุกขั้น
  1. สร้างโฟลเดอร์ใหม่แล้ว git init
  2. สร้างไฟล์ notes.md ใส่ข้อความอะไรก็ได้ → รัน git status (ไฟล์อยู่สถานะ untracked)
  3. git add notes.md → git status (ตอนนี้ staged)
  4. git commit -m "first note" → git log ดู hash ที่ได้
  5. แก้ไฟล์อีกครั้ง → git status (เห็น modified) แล้วลองอธิบายว่าทำไม
เป้าหมาย: เห็นไฟล์เดียวกันเดินทางผ่านสามพื้นที่ครบรอบ

สรุป

  • Git เก็บ snapshot ของทั้งโปรเจกต์ในแต่ละ commit ไม่ใช่ diff
  • งานเดินผ่าน 3 พื้นที่: Working Directory → add → Staging → commit → Repository
  • commit ระบุด้วย hash และชี้ไปยัง parent ทำให้ประวัติเป็นสายโซ่ที่ตรวจสอบได้
  • HEAD บอกว่าคุณยืนอยู่ commit ไหน ทำให้ branch เป็นแค่ pointer ที่เบาและเร็ว
