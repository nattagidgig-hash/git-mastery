บทที่ 4

Merge

เมื่อแตก branch ไปทำงานแล้ว ก็ต้องมีวันรวมกลับ git merge คือการนำงานสองสายมาบรรจบ บทนี้จะเห็นว่า Git ตัดสินใจรวมยังไง และทำไมการ merge ถึง "ต่อ" ประวัติ ไม่ใช่ "ทับ"

🎯 Learning Objectives
  • แยกแยะ fast-forward กับ three-way merge และรู้ว่า Git เลือกแบบไหนเมื่อไหร่
  • เข้าใจ merge commit ที่มี สอง parent และความหมายของมัน
  • ใช้ git merge --no-ff เพื่อรักษาประวัติสายงานให้อ่านออก
  • เห็นชัดว่า merge ไม่ทำลาย commit เดิม — ทุก commit ยังอยู่ครบ

1. Merge คืออะไร

หลังจากเราแตก branch ไปทำ feature สาย main กับสาย feature ก็เดินแยกกัน git merge <branch> คือคำสั่งที่บอก Git ว่า "เอางานจาก branch นั้น มารวมเข้ากับ branch ที่ฉันยืนอยู่ตอนนี้ (HEAD)" จุดสำคัญคือ คุณ merge เข้าหา branch ที่กำลัง checkout อยู่ ไม่ใช่กลับกัน

💡
กฎจำง่าย: ยืนที่ปลายทางก่อน แล้วค่อย merge ต้นทางเข้ามา — เช่นจะเอา feature เข้า main ต้อง git switch main ก่อน แล้วจึง git merge feature

2. สองแบบของการ merge

Git ดูจากตำแหน่งของสองสายแล้วเลือกวิธีรวมให้อัตโนมัติ มีสองแบบหลัก:

3. Fast-forward merge

เกิดขึ้นเมื่อ branch ปลายทางไม่มี commit ใหม่ นับจากจุดที่แตก branch ออกไป กล่าวคือ main ยังอยู่ที่เดิม ส่วน feature เดินไปข้างหน้า เส้นทางจาก main ไป feature เป็นเส้นตรงไม่แตกแขนง Git จึงไม่ต้องสร้าง merge commit ใหม่ — มันแค่ เลื่อน pointer ของ main ไปข้างหน้าให้ตรงกับ feature เท่านั้น

git switch main
git merge feature
Updating b2..d4
Fast-forward          # ไม่มี merge commit — แค่เลื่อน pointer
 readme.txt | 2 +-
✅
Fast-forward สะอาดและไม่มี commit ส่วนเกิน แต่ข้อเสียคือ ประวัติจะมองไม่ออกว่าเคยมี branch แยกออกไป เพราะเส้นเป็นเส้นตรงล้วน

4. Three-way merge

เกิดขึ้นเมื่อ ทั้งสองสายต่างเดินหน้าไปคนละทาง หลังจุดแตก เช่น main มี commit ใหม่ และ feature ก็มี commit ใหม่ด้วย ตอนนี้ไม่มีเส้นตรงให้เลื่อนแล้ว Git จึงต้องดูสาม commit:

  • Base — commit บรรพบุรุษร่วม (จุดที่แตก branch กัน)
  • ปลายของ main — สถานะปัจจุบันของสายที่เรายืนอยู่
  • ปลายของ feature — สถานะที่จะรวมเข้ามา

Git เทียบทั้งสามแล้วรวมการเปลี่ยนแปลง จากนั้นสร้าง merge commit ขึ้นมาใหม่หนึ่งตัว ที่ มี parent สองตัว (ทั้งปลาย main และปลาย feature) นี่คือ commit ชนิดเดียวที่มีมากกว่าหนึ่ง parent

5. fast-forward vs three-way — เทียบกัน

Fast-forwardThree-way merge
เกิดเมื่อปลายทาง (main) ไม่มี commit ใหม่ทั้งสองสายต่างมี commit ใหม่
merge commitไม่มีมี (parent สองตัว)
จำนวน parent—2
รูปร่างประวัติเส้นตรงแตกแขนงแล้วบรรจบ
มองเห็น branch ในกราฟไม่เห็นเห็นชัด
โอกาส conflictไม่มีมีได้ (ถ้าแก้บรรทัดเดียวกัน)

6. --no-ff: บังคับให้มี merge commit เสมอ

บางทีถึงจะ fast-forward ได้ เราก็อยากให้ประวัติ จดจำว่าเคยมี branch แยกออกไป เพื่อให้อ่านง่ายว่า "งานชุดนี้คือ feature เดียวกัน" ใช้ flag --no-ff (no fast-forward) บังคับให้ Git สร้าง merge commit แม้สถานการณ์จะ fast-forward ได้

git switch main
git merge --no-ff feature # บังคับสร้าง merge commit
Merge made by the 'ort' strategy.
✅
หลายทีมตั้งนโยบายให้ merge เข้า main ด้วย --no-ff เสมอ เพราะ merge commit หนึ่งตัวคือ "หมุดหมาย" ที่บอกว่า feature เข้ามาเมื่อไหร่ และ git revert ทั้ง feature ทีเดียวทำได้ง่ายกว่า

7. merge ไม่ทำลายประวัติ

นี่คือหัวใจของบทนี้: merge เป็น operation ที่ เพิ่ม commit เข้าไป (หรือเลื่อน pointer) ไม่เคย ลบหรือเขียนทับ commit เดิมเลย ทุก commit ของทั้งสองสายยังอยู่ใน .git ครบ พร้อม hash เดิม ต่างจาก rebase ที่สร้าง commit ใหม่ทับของเก่า

🛡️
เพราะ merge เป็น "ต่อเติม" ไม่ใช่ "เขียนใหม่" มันจึงปลอดภัยที่จะใช้บน branch ที่แชร์กับคนอื่น (เช่น main ที่ push ขึ้น remote แล้ว) — history ที่คนอื่นมีอยู่ไม่พังเพราะการ merge ของคุณ

8. ลองเอง — Interactive Terminal

เนื่องจาก simulator ทำ merge แบบ fast-forward เราจะลองเคสที่ main ไม่มี commit ใหม่: แตก branch ไปทำงาน แล้ว merge กลับ สังเกตคำว่า Fast-forward

⌨️
พิมพ์ต่อทีละบรรทัด: echo "v2" > app.txt → git add . → git commit -m "work on feature" → git switch main → git merge feature แล้วดูว่ามันรายงาน Fast-forward เพราะ main ไม่ได้ขยับเลยตั้งแต่แตก branch

10. Workshop

🛠️
ภารกิจ: สร้างทั้ง fast-forward และ three-way merge ด้วยตัวเองในเครื่องจริง เพื่อเห็นความต่างของกราฟ
  1. สร้างโฟลเดอร์ใหม่ → git init → สร้าง app.txt commit แรกบน main
  2. git switch -c feature → แก้ไฟล์ → commit หนึ่งครั้ง
  3. git switch main → git merge feature → สังเกตคำว่า Fast-forward
  4. รีเซ็ตการทดลอง: แตก git switch -c feature2 → commit บน feature2 หนึ่งครั้ง
  5. กลับ git switch main → คราวนี้แก้ไฟล์อื่นแล้ว commit บน main ด้วย (ให้ main ขยับ)
  6. git merge feature2 → ตอนนี้จะได้ three-way merge และเกิด merge commit
  7. ปิดท้าย: git log --oneline --graph เทียบรูปร่างกราฟของสองเคส
เป้าหมาย: เห็นด้วยตาว่าเงื่อนไข "main ขยับหรือไม่" คือสิ่งที่ตัดสินว่าจะ fast-forward หรือ three-way

สรุป

  • Fast-forward เกิดเมื่อปลายทางไม่มี commit ใหม่ — Git แค่เลื่อน pointer ไม่มี merge commit
  • Three-way merge เกิดเมื่อสองสายต่างกัน — Git สร้าง merge commit ที่มี 2 parents
  • git merge --no-ff บังคับสร้าง merge commit เพื่อเก็บประวัติว่าเคยมี branch แยก
  • merge ไม่ทำลายประวัติ — เพิ่ม commit เท่านั้น จึงปลอดภัยกับ branch ที่แชร์
  • คุณ merge เข้าหา branch ที่ checkout อยู่เสมอ — ยืนปลายทางก่อน แล้วค่อย git merge <source>
