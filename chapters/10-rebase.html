Rebase

rebase คือการ "ยก" commit ของ branch คุณออกมา แล้วนำไป replay ทับบนปลายใหม่ของอีก branch ผลลัพธ์คือประวัติเป็นเส้นตรงสะอาด ไม่มี merge commit รก ๆ แต่มาพร้อมกฎเหล็กที่ห้ามลืม

🎯 Learning Objectives
  • เข้าใจว่า rebase ต่างจาก merge ยังไง และเลือกใช้ตอนไหน
  • ใช้ git rebase main เพื่อให้ feature branch อยู่บนปลายใหม่ของ main
  • ใช้ interactive rebase (pick/reword/squash/fixup/drop/edit) จัดประวัติให้สะอาด
  • จำ Golden Rule: อย่า rebase commit ที่ push/แชร์ออกไปแล้ว

1. rebase คืออะไร: replay บนปลายใหม่

เวลาคุณแตก feature ออกมาทำงาน แล้วระหว่างนั้น main ก็เดินหน้าไปเรื่อย ๆ ประวัติของคุณจะ "แยกสาย" ออกจากกัน rebase แก้เรื่องนี้ด้วยการ: หา commit ที่เป็นของ feature เท่านั้น → เก็บไว้เป็น patch → ย้ายฐาน (base) ของ feature ไปไว้ที่ปลายล่าสุดของ main → แล้ว replay patch เหล่านั้นทับลงไปทีละอันใหม่

ผลคือประวัติออกมา เป็นเส้นตรง เหมือนคุณเพิ่งเริ่มทำ feature จาก main เวอร์ชันล่าสุดตั้งแต่แรก — ต่างจาก merge ที่ทิ้ง merge commit ไว้เป็นรอยเชื่อมสองสาย

2. ใช้งานจริง: git rebase main

สมมติคุณอยู่บน feature และอยากดึงงานล่าสุดของ main มาเป็นฐาน ขั้นตอนคือ:

git switch feature           # ยืนอยู่บน branch ที่จะถูก replay
git fetch origin              # อัปเดต main ให้เป็นล่าสุดก่อน
git rebase main               # ยก commit ของ feature ไปต่อปลายใหม่ของ main

Successfully rebased and updated refs/heads/feature.

ถ้ามี conflict ระหว่าง replay Git จะหยุดให้คุณแก้ทีละ commit จากนั้น:

git add <ไฟล์ที่แก้แล้ว>
git rebase --continue     # ไปต่อ commit ถัดไป
git rebase --abort        # ยกเลิก กลับสภาพก่อน rebase ทั้งหมด
git rebase --skip         # ข้าม commit ที่กำลังมีปัญหา (ใช้ระวัง)

3. ภาพเคลื่อนไหว: feature ก่อน/หลัง rebase

กดปุ่ม ถัดไป เพื่อดูว่า feature ที่แตกจาก main เก่า ถูกยกไป replay บนปลายใหม่ของ main จนกลายเป็นเส้นตรงได้อย่างไร

💡
สังเกตว่า f1' และ f2' เป็น commit คนละตัวกับของเดิม — เนื้อหาเหมือนเดิมแต่ hash เปลี่ยน เพราะ parent เปลี่ยน นี่คือหัวใจของ Golden Rule ในหัวข้อถัดไป

4. Interactive Rebase: จัดประวัติให้สวย

git rebase -i (interactive) คือเครื่องมือเรียบเรียงประวัติย้อนหลัง เปิดด้วยการระบุว่าจะแก้ย้อนกี่ commit:

git rebase -i HEAD~3       # เปิดแก้ 3 commit ล่าสุด

Git จะเปิด editor พร้อมรายการ commit (เรียงจาก เก่าไปใหม่) ให้คุณเปลี่ยนคำสั่งหน้าแต่ละบรรทัด:

pick   a1b2c3  add login form
squash d4e5f6  fix typo in form
squash 7g8h9i  adjust spacing
# เปลี่ยน 2 บรรทัดล่างเป็น squash → รวมทั้ง 3 เป็น commit เดียว
คำสั่งตัวย่อทำอะไร
pickpใช้ commit นี้ตามเดิม (ค่าเริ่มต้น)
rewordrใช้ commit นี้ แต่ขอแก้ข้อความ message
squashsรวมเข้ากับ commit ก่อนหน้า — เก็บ message ทั้งสองมารวมกัน
fixupfรวมเข้ากับ commit ก่อนหน้า แต่ ทิ้ง message ของตัวเอง
dropdทิ้ง commit นี้ไปเลย (ลบออกจากประวัติ)
editeหยุดที่ commit นี้เพื่อแก้เนื้อหา/แยกออกเป็นหลาย commit
✅
squash เหมาะตอนรวมหลาย commit เล็ก ๆ (เช่น "wip", "fix typo", "อีกที") ให้เป็น commit เดียวที่เล่าเรื่องครบ ก่อนเปิด Pull Request — reviewer จะอ่านง่ายขึ้นมาก

5. ⚠️ Golden Rule

🚫
อย่า rebase commit ที่ push หรือแชร์ออกไปแล้ว
rebase สร้าง commit ใหม่ที่มี hash ต่างจากเดิม ถ้าคนอื่นดึง commit เก่าไปทำงานต่อแล้ว ประวัติของพวกเขาจะ "ชนกัน" กับเวอร์ชันที่คุณ rewrite — ทำให้ history แตก, merge ซ้ำซ้อน, และต้องมานั่งแก้กันยุ่งเหยิง

กฎง่าย ๆ: rebase ได้เฉพาะ commit ที่ยังอยู่ เครื่องคุณคนเดียว (local, ยังไม่ push หรือ push ขึ้น branch ส่วนตัวที่ไม่มีใครใช้) ถ้าจำเป็นต้อง push branch ที่ rebase แล้วจริง ๆ ให้ใช้:

git push --force-with-lease   # ปลอดภัยกว่า --force: ยกเลิกถ้ามีคนอื่น push แทรกมาก่อน

6. ตาราง: merge vs rebase

หัวข้อgit mergegit rebase
รูปทรงประวัติมีกิ่งก้าน + merge commitเส้นตรงสะอาด
เก็บประวัติจริงเก็บลำดับเหตุการณ์จริงครบเขียนใหม่ให้ดูเหมือนทำต่อกัน
hash เดิมของ commitคงเดิมเปลี่ยน (สร้าง commit ใหม่)
ปลอดภัยกับงานที่ push แล้ว✅ ปลอดภัย⚠️ อันตราย (ผิด Golden Rule)
เหมาะกับรวม branch สาธารณะ, เก็บบริบทเก็บ feature ส่วนตัวให้สะอาดก่อนรวม
🧭
แนวทางที่ทีมส่วนใหญ่ใช้: rebase feature ส่วนตัวให้สะอาด ก่อนเปิด PR แล้วใช้ merge ตอนรวมเข้า main เพื่อเก็บร่องรอยว่า PR นี้ถูกรวมเมื่อไหร่

8. Workshop

🛠️
ภารกิจ: squash 3 commit ให้เหลือ commit เดียวที่สะอาด (ทำในเครื่องจริง บน branch ส่วนตัวเท่านั้น)
  1. สร้างโฟลเดอร์ใหม่ → git init → สร้างไฟล์ app.txt แล้ว commit แรกเป็นฐาน
  2. สร้าง 3 commit ต่อกันแบบเล็ก ๆ: แก้ไฟล์แล้ว git commit -m "wip", อีกครั้ง "fix typo", อีกครั้ง "adjust"
  3. รัน git log --oneline ยืนยันว่ามี 3 commit นั้นจริง
  4. รัน git rebase -i HEAD~3 → ในรายการ ปล่อยบรรทัดบนสุดเป็น pick และเปลี่ยนอีก 2 บรรทัดเป็น squash (หรือ fixup ถ้าไม่อยากเก็บ message)
  5. บันทึก editor → ถ้าใช้ squash Git จะให้แก้ message รวม พิมพ์ข้อความที่เล่าเรื่องครบแล้วบันทึก
  6. git log --oneline อีกครั้ง — ยืนยันว่าเหลือ commit เดียว และสังเกตว่า hash เปลี่ยนจากเดิม
เป้าหมาย: เห็นว่า interactive rebase เปลี่ยน 3 commit รก ๆ ให้เหลือ 1 commit สวย ๆ ได้ และเข้าใจว่าทำไมต้องทำก่อน push

สรุป

  • rebase = ยก commit ของ branch ไป replay บนปลายใหม่ของอีก branch → ประวัติ เส้นตรง ไม่มี merge commit
  • git rebase main (ขณะอยู่ feature) ดึงงานล่าสุดของ main มาเป็นฐานใหม่
  • interactive rebase (-i) ใช้ pick/reword/squash/fixup/drop/edit เรียบเรียงประวัติ — squash รวมหลาย commit เป็นหนึ่ง
  • Golden Rule: อย่า rebase commit ที่ push/แชร์แล้ว เพราะ hash เปลี่ยนแล้วทำให้คนอื่นพัง
  • กฎทั่วไป: rebase งานส่วนตัวให้สะอาด, merge งานสาธารณะเพื่อเก็บบริบท
