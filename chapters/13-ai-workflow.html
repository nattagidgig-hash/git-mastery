บทที่ 13

AI Coding Workflow

บทปิดที่ร้อยทุกอย่างเข้าด้วยกัน ในยุคที่ AI ช่วยเขียนโค้ดได้เร็วมาก Git ไม่ได้หมดความสำคัญ แต่กลับสำคัญกว่าเดิม — มันคือ safety net, คือ context ให้ AI, และคือวินัยที่ทำให้คุณยังควบคุมโค้ดตัวเองได้

🎯 Learning Objectives
  • วาง workflow แบบ spec-driven — ให้ PROJECT.md เป็น single source of truth ก่อนปล่อย AI ลงมือ
  • ใช้ commit เล็กและ atomic + branch ต่อ experiment เพื่อ review/revert งานของ AI ได้เป็นจุด ๆ
  • ใช้ Git เป็น safety net เมื่อ AI แก้พัง: revert / reflog / reset
  • เชื่อมทุกบทเข้าลูป Spec → Architecture → Planning → Implementation → Review → Refactor → Release

1. ทำไม Git ยิ่งสำคัญในยุค AI

AI agent เขียนโค้ดได้เร็วและเยอะ — แต่มันก็แก้พังได้เร็วและเยอะเช่นกัน ปัญหาไม่ใช่ "AI เขียนถูกไหม" แต่คือ "ถ้าผิด คุณย้อนกลับได้ไหม และคุณ review ทันหรือเปล่า" Git ตอบทั้งสองข้อ: ทุก commit คือจุด restore, ทุก branch คือ sandbox ที่ทิ้งได้, และ history คือ context ที่ AI รุ่นถัดไปอ่านเข้าใจ

💡
กรอบคิดหลัก: AI คือคนเขียนโค้ดที่เร็วมากแต่ไม่จำอะไรข้ามรอบ — Git คือ "ความจำระยะยาว" และ "ปุ่ม undo" ของทั้งทีม คุณกับ AI ใช้ร่วมกัน

2. ลูป Workflow: Spec → … → Release

การทำงานกับ AI ที่ได้ผลไม่ใช่ "บอกให้เขียนทั้งระบบทีเดียว" แต่เป็นการเดินเป็นลูปสั้น ๆ ซ้ำ ๆ แต่ละรอบจบด้วยสถานะที่ commit ได้ เว็บ Git Mastery ที่คุณกำลังเรียนอยู่นี้ก็สร้างด้วยลูปเดียวกัน — เริ่มจาก PROJECT.md เป็น spec ก่อนเขียนโค้ดสักบรรทัด

ลูกศรสีเน้น (accent) ช่วงท้ายคือการ "วนกลับ" — Release แล้วก็กลับไปอัปเดต Spec รอบใหม่ ไม่มีจุดจบ มีแต่รอบถัดไป

3. Spec-driven: PROJECT.md ก่อนเสมอ

AI ทำงานดีที่สุดเมื่อมี "เป้าหมายเดียวที่ชัด" ก่อนให้มันเขียนโค้ด ให้เขียน spec ลงไฟล์ก่อน (เช่น PROJECT.md) แล้ว commit ขึ้น repo ไฟล์นี้กลายเป็น single source of truth ที่ทั้งคุณ, เพื่อนร่วมทีม, และ AI อ้างอิงตรงกัน — เมื่อ AI เริ่มหลงทาง คุณชี้กลับมาที่ spec ได้

git add PROJECT.md
git commit -m "docs: define spec — 13 บท + tech stack (no build step)"
# commit แรกของฟีเจอร์ = สัญญาว่าจะทำอะไร ไม่ใช่โค้ด
📜
โปรเจกต์เว็บนี้เริ่มจาก PROJECT.md ที่ list ครบ 13 บท + design style + tech ก่อนเขียน HTML สักไฟล์ นั่นคือ spec-driven ของจริง

4. Commit เล็ก + atomic = ของขวัญให้ AI และตัวคุณ

"Atomic commit" คือ commit ที่ทำเรื่องเดียวจบในตัว ถ้า AI สร้างฟีเจอร์ใหญ่ ให้แตกเป็น commit ย่อย ๆ ทีละขั้น ข้อดี: (1) คุณ review ได้เป็นจุด ไม่ต้องอ่าน diff 2,000 บรรทัดรวด, (2) ถ้าขั้นไหนพัง git revert เฉพาะ commit นั้นได้โดยไม่ลากของดีไปด้วย, (3) git log ที่อ่านรู้เรื่องกลายเป็น context ให้ AI agent รอบถัดไปเข้าใจว่า "ทำไมโค้ดเป็นแบบนี้"

✅
เคล็ดลับ: commit เล็ก = หน่วย review ที่เล็ก = AI (และมนุษย์) review แม่นขึ้น ลองตั้งกฎ "1 commit = 1 เหตุผล" ถ้าอธิบายไม่ได้ในประโยคเดียว แปลว่ามันใหญ่ไป ควรแตก

เทียบ commit ที่ดีกับไม่ดี:

❌ Commit ก้อนใหญ่✅ Commit เล็ก atomic
Reviewdiff 1,500 บรรทัด อ่านไม่ไหว ปล่อยผ่านอ่านได้ทีละ 30–80 บรรทัด เห็น bug ง่าย
Revertย้อนได้ทั้งก้อน ทิ้งของดีไปด้วยgit revert <hash> เฉพาะส่วนที่พัง
Context ให้ AI"update stuff" — AI เดาเจตนาไม่ออกmessage อธิบาย why → AI ต่อยอดถูก
Bisect หา bugหาไม่เจอว่า commit ไหนทำพังgit bisect ชี้ commit ต้นเหตุได้

5. Branch ต่อ feature/experiment: ลองของ AI แบบทิ้งได้

กฎเหล็กในการทำงานกับ AI: อย่าให้ AI แก้ตรงบน main ให้เปิด branch แยกต่อทุก feature หรือทุกการทดลอง ถ้า AI ทำออกมาดี → merge เข้า main; ถ้าไม่ดี → ลบทั้ง branch ทิ้ง main ไม่เคยแปดเปื้อน นี่ทำให้คุณ "ลองของเสี่ยง" กับ AI ได้อย่างสบายใจ

กด prev/next เพื่อดูทีละขั้น ถ้า experiment ไม่เวิร์ก: git branch -D ai/try-feature — จบ ไม่ต้อง revert อะไรบน main เลย

6. Pull Request + review (มนุษย์ + AI)

ก่อน merge ทุกครั้งให้ผ่าน Pull Request เพื่อให้มี "ด่านตรวจ" — ทั้งคนและ AI review code ได้ บน PR คุณเห็น diff รวม, รัน CI, และ comment เป็นบรรทัดได้ การให้ AI review PR ของ AI อีกตัว (หรือ review ของมนุษย์) จับ bug ที่คนเขียนมองข้ามได้บ่อย

git switch -c ai/try-feature
# ... AI ทำงาน, commit เล็ก ๆ ...
git push -u origin ai/try-feature
gh pr create --fill          # เปิด PR ให้คน + AI review ก่อน merge
https://github.com/you/git-mastery/pull/42
🔍
ให้ AI ช่วย review แต่ มนุษย์เป็นคนกด merge เสมอ — AI เสนอได้ คุณตัดสินใจ นี่คือเส้นแบ่งความรับผิดชอบที่ปลอดภัยที่สุด

7. Git เป็น safety net เมื่อ AI แก้พัง

เมื่อ AI แก้แล้วพัง คุณมีสามทางหลัก เลือกตามสถานการณ์ (ทบทวนบทที่ 8–11 ได้):

คำสั่งทำอะไรใช้เมื่อ
git revert <hash>สร้าง commit ใหม่ที่ "ยกเลิก" commit เก่า (ไม่ลบประวัติ)commit นั้น push/แชร์ไปแล้ว — ปลอดภัยสุด
git reset --hard <hash>ย้าย branch กลับไปจุดเดิม ทิ้งงานหลังจากนั้นงานยังอยู่ในเครื่องคุณคนเดียว ยังไม่ push
git reflogดู log ของทุกการขยับ HEAD — กู้ commit ที่ "หาย" ได้เผลอ reset/rebase ทิ้งของสำคัญ
 
git reflog                       # เห็นทุกจุดที่ HEAD เคยอยู่ แม้ commit ที่ดูเหมือนหาย
e5a1b2c HEAD@{0}: reset: moving to HEAD~3
9f3c1a0 HEAD@{1}: commit: ai: add feature (งานที่นึกว่าหาย)
git reset --hard 9f3c1a0       # กู้กลับมาจาก reflog ได้
⚠️
git reset --hard ทิ้งงานใน Working Directory ที่ยังไม่ commit แบบกู้ไม่ได้ ถ้า AI กำลังแก้ค้างอยู่ ให้ git stash หรือ commit ก่อนเสมอ บน history ที่ push แล้ว ให้ใช้ revert ไม่ใช่ reset

8. Commit message ที่ดี = context ให้ AI agent

AI agent รุ่นถัดไปไม่ได้นั่งอยู่ในหัวคุณ มันอ่าน git log เพื่อเข้าใจว่าโค้ดวิวัฒน์มายังไง message ที่เขียน "why" (ไม่ใช่แค่ "what") กลายเป็นเอกสารที่ AI ใช้ต่อยอดได้ทันที ลองใช้แนวทาง Conventional Commits: type: subject

# ❌ ไม่ดี — ไม่บอกเจตนา
fix stuff

# ✅ ดี — type + why ชัด AI/คนอ่านต่อได้
fix(terminal): escape user input ก่อน render กัน XSS ใน simulator

9. .gitignore, Semantic Versioning + Tag Release

ปิดท้ายลูปที่ Release ให้ .gitignore กันไฟล์ที่ไม่ควรเข้า repo (เช่น node_modules/, .env, ไฟล์ที่ AI tool สร้างชั่วคราว) แล้วใช้ Semantic Versioning (MAJOR.MINOR.PATCH) ติด tag ที่จุด release เพื่อให้ย้อนกลับไปเวอร์ชันที่ "ใช้งานได้จริง" ได้ทุกเมื่อ

# .gitignore
node_modules/
.env
.DS_Store
*.log

git tag -a v1.0.0 -m "release: ครบ 13 บท พร้อมใช้งาน"  # MAJOR.MINOR.PATCH
git push origin v1.0.0
# MAJOR=เปลี่ยนจนพังของเดิม · MINOR=เพิ่มฟีเจอร์ · PATCH=แก้ bug

11. Workshop

🛠️
ภารกิจ — mini-cycle ครบลูป: จำลองการทำงานกับ AI หนึ่งรอบ ตั้งแต่ branch จนถึง merge พร้อมทดสอบ safety net ลองใน terminal จริงในเครื่องคุณ
  1. ใน repo ฝึก: git switch -c ai/experiment (แตก branch ให้ AI ลองงาน — main ปลอดภัย)
  2. commit เล็ก ๆ สองก้อน: แก้ทีละเรื่องแล้ว git commit -m "..." ทั้งสองครั้ง (atomic)
  3. ทดสอบ safety net: commit ที่ "พัง" หนึ่งก้อน แล้ว git revert HEAD — สังเกตว่าได้ commit ใหม่ที่ยกเลิกของเดิมโดยไม่ลบประวัติ
  4. กลับไป main: git switch main แล้ว git merge ai/experiment (review ก่อน merge ในใจ: diff อ่านรู้เรื่องไหม?)
  5. ปิดลูป: git tag -a v0.1.0 -m "first working cut" แล้ว git log --oneline --graph ดูเส้นทางทั้งหมด
เป้าหมาย: รู้สึกถึงจังหวะ branch → commit เล็ก → revert (safety net) → merge → tag — ลูปเดียวกับที่ใช้สร้างเว็บ Git Mastery นี้

สรุป

  • เริ่มจาก spec (เช่น PROJECT.md) เป็น single source of truth ก่อนปล่อย AI ลงมือ
  • commit เล็ก atomic + branch ต่อ experiment = review/revert เป็นจุด และให้ context กับ AI
  • Pull Request + review (คน + AI) ก่อน merge เสมอ — AI เสนอ มนุษย์ตัดสินใจ
  • Git คือ safety net: revert สำหรับของที่ push แล้ว, reset สำหรับงานในเครื่อง, reflog กู้ของที่หาย
  • ปิดลูปด้วย .gitignore + semver + tag แล้ววนกลับไป Spec รอบใหม่
