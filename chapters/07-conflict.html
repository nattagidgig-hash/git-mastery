Merge Conflict

conflict ไม่ใช่ error และไม่ใช่ความผิดของคุณ — มันคือ Git บอกตรง ๆ ว่า "สองฝั่งแก้ที่เดียวกันคนละแบบ ฉันตัดสินใจแทนไม่ได้ ช่วยตัดสินให้หน่อย" เมื่อเข้าใจ markers และวงจรแก้ คุณจะเลิกกลัวมัน

🎯 Learning Objectives
  • เข้าใจว่า merge conflict เกิดขึ้นเมื่อไร และทำไม Git ตัดสินใจแทนไม่ได้
  • อ่าน conflict markers จริง (<<<<<<< / ======= / >>>>>>>) ได้
  • แก้ conflict ครบวงจร: เลือก/รวมโค้ด → ลบ markers → add → commit
  • รู้จัก ours/theirs, mergetool และวิธี git merge --abort เพื่อถอยกลับ

1. conflict เกิดขึ้นเมื่อไร

เวลา merge สอง branch เข้าด้วยกัน Git พยายามรวมการเปลี่ยนแปลงให้อัตโนมัติ ถ้าสองฝั่งแก้ไฟล์คนละจุด (คนละบรรทัด คนละไฟล์) Git รวมให้เองได้สบาย ไม่มีปัญหา

แต่ถ้าสองฝั่ง แก้บรรทัดเดียวกัน ให้เป็นเนื้อหาที่ต่างกัน Git จะไม่เดาว่าใครถูก เพราะถ้าเดาผิดอาจทำโค้ดพังเงียบ ๆ มันจึงหยุด แล้วโยน merge conflict ออกมาให้คุณเป็นคนตัดสิน

💡
ใจความ: conflict ไม่ได้แปลว่าทำผิด มันคือกลไกความปลอดภัย — Git ยอมหยุดให้คุณตัดสินใจ ดีกว่ารวมโค้ดมั่ว ๆ แล้วพังทีหลัง

2. หน้าตา conflict markers ของจริง

เมื่อ merge แล้วชน Git จะแก้ไฟล์ที่ขัดแย้งให้ โดยใส่ conflict markers ครอบโค้ดทั้งสองฝั่งไว้ในไฟล์เดียวกัน หน้าตาแบบนี้:

# config.py — หลัง git merge feature แล้วชน
<<<<<<< HEAD
price = 120        # โค้ดฝั่งเรา (branch ที่กำลังยืนอยู่ = ours)
=======
price = 150        # โค้ดฝั่งเขา (branch ที่ดึงเข้ามา = theirs)
>>>>>>> feature

อ่านง่าย ๆ เป็นสามส่วน:

Markerหมายถึง
<<<<<<< HEADเริ่มฝั่ง ของเรา (ours) — branch ที่คุณยืนอยู่ตอนสั่ง merge
=======เส้นแบ่งกลาง ระหว่างของเรากับของเขา
>>>>>>> featureจบฝั่ง ของเขา (theirs) — branch ที่คุณกำลังดึงเข้ามา (ชื่อหลัง >>>)
⚠️
ห้ามเด็ดขาด: อย่า git add / git commit ทั้งที่ <<<<<<<, =======, >>>>>>> ยังค้างอยู่ในไฟล์ Git จะไม่เตือน แต่ markers จะกลายเป็นโค้ดจริงที่ทำโปรแกรมพัง หรือ syntax error

3. วงจรแก้ conflict

ตอนชน Git จะหยุดกลางคัน รัน git status จะเห็นว่าไฟล์ไหนขัดแย้ง (อยู่ใต้หัวข้อ Unmerged paths):

git merge feature
Auto-merging config.py
CONFLICT (content): Merge conflict in config.py
Automatic merge failed; fix conflicts and then commit the result.

git status
Unmerged paths:
  (use "git add <file>..." to mark resolution)
        both modified:   config.py

ขั้นตอนแก้ มี 4 step เสมอ:

  1. เปิดไฟล์ ที่ขัดแย้ง หา markers
  2. ตัดสินใจ — เก็บฝั่งเรา, เก็บฝั่งเขา, หรือ รวมทั้งสองฝั่งเป็นเนื้อหาใหม่
  3. ลบ markers ทั้งสามเส้น ออกให้หมด เหลือแต่โค้ดที่ถูกต้อง
  4. git add ไฟล์นั้น (= บอก Git ว่าแก้เสร็จแล้ว) แล้ว git commit เพื่อปิดงาน merge

หลังแก้ไฟล์ config.py ให้เหลือบรรทัดเดียวที่ถูกต้อง (สมมติรวมเป็น price = 150) ก็ปิดงาน:

git add config.py          # mark ว่า resolve แล้ว
git commit               # commit merge (Git เติม message ให้อัตโนมัติ)
[main 9f3c1a2] Merge branch 'feature'
✅
commit แรกหลัง resolve คือ merge commit ที่มี 2 parents (ฝั่งเรากับฝั่งเขา) — มันคือจุดที่สองสายมาบรรจบกันอย่างเป็นทางการ

4. ก่อน/หลัง: ไฟล์เปลี่ยนจากอะไรเป็นอะไร

เทียบให้เห็นภาพชัด ๆ ว่าการ "แก้ conflict" คือการเปลี่ยนกองมาร์กเกอร์ ให้กลายเป็นโค้ดที่เราตั้งใจ:

ก่อนแก้ (Git ใส่ marker ให้)หลังแก้ (เราตัดสินใจ + ลบ marker)
<<<<<<< HEAD
price = 120
=======
price = 150
>>>>>>> feature		 price = 150
(เลือกฝั่ง theirs และลบ marker ครบ)

5. ถ้าอยากถอย: git merge --abort

ถ้า conflict ดูยุ่งเกินไป หรือ merge ผิด branch แค่อยากกลับไปสภาพ ก่อนเริ่ม merge ทั้งหมด ใช้:

git merge --abort     # ยกเลิก merge ทิ้ง กลับไปจุดก่อนสั่ง merge เป๊ะ ๆ

คำสั่งนี้จะทิ้งงาน merge ที่ค้างอยู่ทั้งหมด ไฟล์ทุกอันกลับเป็นเหมือนตอนก่อน git merge ปลอดภัยเสมอ ตราบใดที่ยังไม่ได้ commit merge ออกไป

🧭
เคล็ดลับ: ถ้าเริ่มแก้ conflict แล้วงงว่าแก้ไปถึงไหน git merge --abort คือปุ่ม "เริ่มใหม่" ที่ปลอดภัยที่สุด

6. ทางลัด: ours / theirs และ mergetool

บางครั้งคุณรู้แน่ ๆ ว่าจะเอาทั้งฝั่งของฝั่งใดฝั่งหนึ่ง ไม่ต้องเปิดแก้ด้วยมือ ใช้ git checkout เลือกฝั่งของไฟล์ที่ชนได้:

git checkout --ours config.py     # เอาฝั่งเรา (branch ปัจจุบัน = HEAD) ทั้งไฟล์
git checkout --theirs config.py   # เอาฝั่งเขา (branch ที่ merge เข้ามา) ทั้งไฟล์
git add config.py               # แล้ว mark ว่า resolve เสร็จ
⚠️
--ours / --theirs เอาทั้งไฟล์ของฝั่งนั้น ไม่ใช่ทีละบรรทัด ถ้าไฟล์มีหลายจุดที่อยากเก็บคนละฝั่ง วิธีนี้จะทิ้งของอีกฝั่งหมด — กรณีนั้นต้องแก้ด้วยมือ

ถ้า conflict ใหญ่ ใช้เครื่องมือ visual ช่วยเทียบสามทาง (ours / base / theirs) ได้ด้วย git mergetool ซึ่งจะเปิดโปรแกรมเทียบไฟล์ที่ตั้งค่าไว้ (เช่น VS Code, Beyond Compare, vimdiff):

git mergetool             # เปิด GUI/editor เทียบไฟล์ทีละจุดที่ชน

เครื่องมือสมัยใหม่อย่าง VS Code ยังแสดงปุ่ม Accept Current (ours) / Accept Incoming (theirs) / Accept Both เหนือ marker แต่ละก้อน ช่วยให้กดเลือกได้โดยไม่ต้องลบ marker เอง

7. ลำดับเหตุการณ์ตอนเกิด conflict

มองภาพรวมเป็นกราฟ: สองสายแยกจาก base เดียวกัน ต่างแก้บรรทัดเดียวกัน แล้วมาบรรจบที่ merge commit หลัง resolve

9. Workshop

🛠️
ภารกิจ: สร้าง conflict ขึ้นมาเองแล้วแก้ให้จบ — ทำใน terminal จริงในเครื่อง (เพราะ conflict เกี่ยวกับการแก้ไฟล์ด้วยมือ ตัวจำลองทำไม่ได้)
  1. สร้างโฟลเดอร์ใหม่ → git init → สร้างไฟล์ price.txt ใส่บรรทัด price = 100
  2. git add price.txt → git commit -m "base"
  3. บน main แก้เป็น price = 120 → git commit -am "main change"
  4. สร้าง branch จาก base: git switch -c feature HEAD~1 → แก้เป็น price = 150 → git commit -am "feature change"
  5. กลับ main: git switch main → git merge feature → ควรเห็น CONFLICT
  6. เปิด price.txt สังเกต markers ทั้งสามเส้น → เลือก/รวมค่า → ลบ markers ให้หมด
  7. git add price.txt → git commit → git log --oneline ดู merge commit ที่ได้
  8. โบนัส: ลองทำใหม่อีกรอบ แต่คราวนี้ตอนชนให้สั่ง git merge --abort สังเกตว่าไฟล์กลับเป็นยังไง
เป้าหมาย: เห็นวงจร CONFLICT → แก้มือ → add → commit ครบ และรู้ทางถอยด้วย --abort

สรุป

  • conflict เกิดเมื่อสอง branch แก้ บรรทัด/ส่วนเดียวกัน ให้ต่างกัน — Git ไม่เดาแทน
  • markers สามเส้น: <<<<<<< HEAD (ours) / ======= / >>>>>>> branch (theirs)
  • วงจรแก้: เปิดไฟล์ → เลือก/รวมโค้ด → ลบ markers → git add → git commit
  • อย่า commit ทั้งที่ markers ยังอยู่ — Git ไม่เตือน แต่โค้ดจะพัง
  • git merge --abort ถอยกลับสภาพก่อน merge ได้เสมอ; --ours/--theirs และ mergetool เป็นทางลัด
