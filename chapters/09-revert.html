Revert

เวลา commit เสียหลุดขึ้น branch ที่ทีมแชร์กันแล้ว คุณ "ลบ" มันทิ้งเฉย ๆ ไม่ได้ — ต้องยกเลิกแบบที่ทุกคนตามทันและประวัติไม่พัง นี่คือหน้าที่ของ git revert

🎯 Learning Objectives
  • เข้าใจว่า git revert สร้าง commit ใหม่ที่กลับด้าน ของ commit เป้าหมาย โดยประวัติเดิมยังอยู่ครบ
  • รู้ว่าทำไม revert ถึง ปลอดภัยกับ history ที่ push/แชร์แล้ว ในขณะที่ reset ไม่ปลอดภัย
  • เปรียบเทียบ reset vs revert และเลือกใช้ให้ถูกสถานการณ์
  • revert merge commit ด้วย flag -m ได้อย่างถูกต้อง

1. revert ทำอะไร: "เพิ่ม" ไม่ใช่ "ลบ"

git revert <commit> ไม่ได้ลบ commit ที่ผิดออกจากประวัติ แต่จะคำนวณการเปลี่ยนแปลงของ commit นั้นแล้ว สร้าง commit ใหม่ที่ทำตรงกันข้าม ต่อท้ายเข้าไป ถ้า commit เดิมเพิ่มบรรทัด revert ก็จะลบบรรทัดนั้น ถ้าเดิมลบไฟล์ revert ก็เอาไฟล์กลับมา ผลลัพธ์สุดท้ายเหมือน commit ที่ผิดไม่เคยเกิดขึ้น แต่ ร่องรอยทั้งหมดยังอยู่ในประวัติ

💡
คิดง่าย ๆ: reset = ย้อนเวลา (ลบบรรทัดในไดอารี) · revert = เขียนบรรทัดใหม่ว่า "ขอยกเลิกที่เขียนไปเมื่อวาน" ไดอารีหน้าเดิมยังอยู่ครบ

2. ใช้งานพื้นฐาน

เลือก commit ที่อยากยกเลิกแล้วสั่ง revert ได้เลย Git จะเปิด editor ให้ใส่ข้อความ commit (ค่าเริ่มต้นคือ Revert "...") จากนั้นจะมี commit ใหม่โผล่บนยอด branch

git log --oneline
c3d4e5f (HEAD -> main) add checkout button
b2c3d4e break login form        # <- commit ที่ผิด
a1b2c3d init project

git revert b2c3d4e          # ยกเลิกเฉพาะ commit นี้
[main 9f8e7d6] Revert "break login form"
 1 file changed, 4 insertions(+), 4 deletions(-)

git log --oneline
9f8e7d6 (HEAD -> main) Revert "break login form"  # commit ใหม่
c3d4e5f add checkout button
b2c3d4e break login form                            # ยังอยู่!
a1b2c3d init project
⚠️
คำสั่ง git revert, git reset, git push และอื่น ๆ ที่นอกเหนือชุดพื้นฐาน terminal จำลองในเว็บนี้ยังไม่รองรับ ให้ลองในเครื่องจริงตาม workshop ท้ายบท

3. ทำไมต้องใช้ revert บน shared branch

หัวใจอยู่ตรงนี้: เมื่อคุณ push commit ขึ้น remote แล้ว เพื่อนร่วมทีมอาจ pull ไปแล้ว — commit นั้นกลายเป็น ประวัติสาธารณะ ที่หลายคนถืออยู่ ถ้าคุณใช้ reset เพื่อ "ลบ" มัน ประวัติบนเครื่องคุณกับบน remote จะแตกกัน เวลา push ต้องใช้ --force ซึ่งจะไปเขียนทับประวัติของคนอื่น ทำให้ commit ของเพื่อนหายหรือ history พังกันทั้งทีม

revert แก้ปัญหานี้เพราะมันแค่ เพิ่ม commit ใหม่ต่อท้าย ไม่แตะของเดิม ทุกคนแค่ pull ตามปกติก็ได้ commit ยกเลิกไปด้วย ไม่มีใครต้องแก้ประวัติในเครื่องตัวเอง

4. reset vs revert

ทั้งคู่ใช้ "ย้อน" งาน แต่กลไกคนละแบบและเหมาะกับคนละสถานการณ์

git resetgit revert
กลไกย้าย HEAD/branch ถอยหลัง — เขียนประวัติใหม่เพิ่ม commit ใหม่ ที่กลับด้าน
ประวัติเดิมหายไป (commit หลุดออกจากสาย)ยังอยู่ครบ
ปลอดภัยกับ public history?❌ ไม่ — ต้อง push --force พังของทีม✅ ใช่ — push ปกติ ทุกคน pull ตาม
เหมาะกับcommit ที่ยัง local ยังไม่ pushcommit ที่ push/แชร์แล้ว
ผลกับ working dirมี --soft/--mixed/--hard ให้เลือกเปลี่ยนไฟล์ผ่าน commit ใหม่
✅
กฎง่าย ๆ: ถ้า commit ยังอยู่แค่ในเครื่องเรา ใช้ reset ได้ ถ้ามัน push ขึ้นไปแล้ว/แชร์กับทีมแล้ว ใช้ revert เสมอ

5. revert merge commit ต้องใช้ -m

merge commit มี parent สองตัว (สาย main เดิม + สาย feature ที่ถูก merge เข้ามา) Git จึงไม่รู้เองว่าจะให้ "กลับด้าน" โดยอ้างอิงสายไหนเป็นหลัก คุณต้องบอกด้วย -m ตามด้วย เลข parent (เริ่มที่ 1) ที่จะใช้เป็น mainline

git revert -m 1 <merge-commit>   # เก็บสาย parent ที่ 1 (มักเป็น main) ยกเลิกสิ่งที่ feature เพิ่มเข้ามา

โดยทั่วไป -m 1 หมายถึง "ยกเลิกการเปลี่ยนแปลงทั้งหมดที่ branch ที่ถูก merge เข้ามานำมา" ส่วน -m 2 จะอ้างอิงอีกสายแทน เลือกผิด = revert ผิดสาย

⚠️
ข้อควรระวัง: หลัง revert merge ไปแล้ว ถ้าอยากเอา feature กลับเข้ามาอีกครั้ง คุณ merge สาย feature เดิมซ้ำตรง ๆ ไม่ได้ (Git มองว่า merge ไปแล้ว) ต้อง "revert ตัว revert" หรือ rebase feature ใหม่ — ทางที่ดีที่สุดคือเลี่ยงปัญหาด้วยการ revert ให้น้อยครั้งและเข้าใจก่อนทำ

7. Workshop

🛠️
ภารกิจ: จำลองการ "ยกเลิก commit ที่หลุดขึ้นไปแล้ว" ด้วย revert ในเครื่องจริง สังเกต git log --oneline ทุกขั้น
  1. สร้างโฟลเดอร์ใหม่ → git init → สร้างไฟล์ app.txt ใส่ v1 แล้ว git add . && git commit -m "init"
  2. แก้ app.txt เป็น v2 (buggy) → git commit -am "buggy change"
  3. git log --oneline จด hash ของ commit "buggy change" ไว้
  4. git revert <hash> (กด save/close ใน editor เพื่อรับข้อความ commit เริ่มต้น)
  5. เปิด app.txt ดู → เนื้อหากลับเป็น v1 แล้ว แต่ git log --oneline ยังเห็นทั้ง "buggy change" และ "Revert ..." ครบ
  6. โบนัส: ลอง git revert HEAD อีกครั้ง (revert ตัว revert) แล้วดูว่าเนื้อหากลับไปเป็น v2 อีกที — เข้าใจว่าทุกการยกเลิกคือ commit ใหม่เสมอ
เป้าหมาย: เห็นว่า revert ทำให้ "ยกเลิกได้โดยไม่ลบประวัติ" และทุกขั้นย้อนกลับได้

สรุป

  • git revert <commit> สร้าง commit ใหม่ที่กลับด้าน ของ commit เป้าหมาย — ไม่ลบของเดิม ประวัติยังครบ
  • ปลอดภัยกับ history ที่ push/แชร์แล้ว เพราะแค่เพิ่ม commit ต่อท้าย push ปกติ ทุกคน pull ตามได้
  • reset เขียนประวัติใหม่ — ใช้ได้กับ commit ที่ยัง local เท่านั้น ห้ามใช้กับ commit สาธารณะ
  • revert merge commit ต้องระบุ mainline ด้วย -m <เลข parent> (มักเป็น -m 1)
