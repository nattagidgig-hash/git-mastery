บทที่ 2

Commit

commit คือการกด "ถ่ายภาพ" โปรเจกต์แล้วบันทึกถาวร แต่ก่อนถ่าย คุณต้องจัดฉากก่อนด้วย git add บทนี้จะปูวงจรไฟล์ตั้งแต่ untracked จนถึง committed และสอนเขียน commit ที่ดีที่อนาคต(และทีม)จะขอบคุณ

🎯 Learning Objectives
  • ใช้ git add แบบทีละไฟล์, ทั้งหมด (.) และเลือกทีละส่วน (-p)
  • เข้าใจวงจร untracked → staged → committed → modified ผ่าน git status
  • แยก git diff (working vs commit ล่าสุด) ออกจาก git diff --staged
  • เขียน commit message ที่ดี + ทำ atomic commit และแก้ commit ล่าสุดด้วย --amend

1. commit คืออะไรกันแน่

commit คือการบันทึก snapshot ของไฟล์ที่อยู่ใน Staging Area ลงไปใน repository อย่างถาวร แต่ละ commit เก็บ: snapshot ของไฟล์, ชื่อ/อีเมลผู้เขียน, เวลา, ข้อความอธิบาย (message) และ pointer ไปยัง commit ก่อนหน้า (parent) จากบทที่แล้วเรารู้ว่า Git ไม่ได้เก็บ diff แต่เก็บภาพถ่ายทั้งชุด — commit คือปุ่มชัตเตอร์นั้นเอง

💡
commit ไม่ได้บันทึก "ทุกอย่างใน working directory" แต่บันทึก เฉพาะสิ่งที่อยู่ใน staging area เท่านั้น นี่คือเหตุผลที่ต้อง git add ก่อนเสมอ

2. วงจรชีวิตของไฟล์ (File Lifecycle)

ทุกไฟล์ในสายตา Git อยู่ในสถานะใดสถานะหนึ่ง การเข้าใจการเดินทางนี้ทำให้อ่าน git status ออกทันที

สถานะหมายความว่าgit status แสดง
Untrackedไฟล์ใหม่ที่ Git ยังไม่เคยเห็นUntracked files
Stagedถูก add แล้ว รอเข้า commit หน้าChanges to be committed
Committed / Unmodifiedบันทึกใน repo แล้วและยังไม่ถูกแก้(ไม่แสดง — clean)
Modifiedเคย commit แล้วแต่ตอนนี้แก้ใหม่ ยังไม่ stageChanges not staged for commit

3. git add: จัดของเข้า staging

git add ย้ายการเปลี่ยนแปลงจาก working directory เข้าสู่ staging area คุณควบคุมได้ว่าจะใส่อะไรเข้าไปบ้าง

git add app.js                 # stage ทีละไฟล์ — แม่นยำที่สุด
git add src/                   # stage ทั้งโฟลเดอร์
git add .                       # stage ทุกอย่างในโฟลเดอร์ปัจจุบันลงไป
git add -p                      # เลือก stage ทีละ "hunk" (ทีละส่วนในไฟล์)

git add -p (ย่อจาก --patch) จะถามคุณทีละก้อนการเปลี่ยนแปลงว่า "stage ก้อนนี้ไหม?" ตอบ y = ใช่, n = ไม่, s = แยกก้อนให้เล็กลง, q = ออก เครื่องมือนี้คือกุญแจสู่ atomic commit เพราะให้คุณแยกการแก้หลายเรื่องในไฟล์เดียวออกเป็นคนละ commit ได้

⚠️
git add . สะดวกแต่เสี่ยง — มันจะ stage ทุกไฟล์รวมถึงที่คุณไม่ตั้งใจ (ไฟล์ทดสอบ, log, secret) เช็ก git status ก่อน commit เสมอ และอย่าลืมตั้ง .gitignore

4. git commit: กดชัตเตอร์

เมื่อจัด staging เสร็จ ก็บันทึกด้วย git commit ใส่ message ด้วย -m ถ้าไม่ใส่ Git จะเปิด editor ให้พิมพ์

git commit -m "Add login form validation"
git commit                       # ไม่ใส่ -m จะเปิด editor ให้พิมพ์ message หลายบรรทัด
git commit -am "Fix typo"      # -a = add ไฟล์ที่ tracked แล้วทั้งหมดให้อัตโนมัติ (ข้าม untracked)
📝
git commit -a จะ stage เฉพาะไฟล์ที่ เคย tracked แล้ว เท่านั้น ไฟล์ใหม่ (untracked) จะไม่ถูกใส่ — ต้อง git add เองก่อน

5. ลองเอง — Interactive Terminal

นี่คือ git CLI จำลอง ลองวงจรเต็ม: สร้างไฟล์ → add → commit → ดู log สังเกตว่า git status เปลี่ยนทุกขั้น

⌨️
ลองต่อ: echo "console.log(2)" > app.js แล้ว git diff เพื่อดูความต่าง จากนั้น git add app.js แล้ว git diff --staged — สังเกตว่าผลลัพธ์สองคำสั่งต่างกัน

6. git diff vs git diff --staged

คำสั่งสองตัวนี้ดูคล้ายแต่เปรียบเทียบคนละคู่ ใช้ผิดจะงงว่า "ทำไมไม่เห็นการเปลี่ยนแปลง" จุดสำคัญคือ staging area อยู่ตรงกลาง ระหว่าง working directory กับ commit ล่าสุด

คำสั่งเปรียบเทียบตอบคำถามว่า
git diffWorking Directory ↔ Staging Area"แก้อะไรไปบ้างที่ยังไม่ได้ add?"
git diff --staged
(= --cached)		Staging Area ↔ commit ล่าสุด"กำลังจะ commit อะไรไป?"
git diff HEADWorking Directory ↔ commit ล่าสุด"แก้อะไรไปบ้างทั้งหมด (รวม staged + unstaged)?"
✅
เคยเจอไหม? แก้ไฟล์แล้ว git add ไปแล้ว พอรัน git diff กลับว่าง — ไม่ใช่บั๊ก! เพราะ working dir กับ staging ตรงกันแล้ว ความต่างไปโผล่ที่ git diff --staged แทน

7. Commit message ที่ดี

commit message คือจดหมายถึงตัวคุณในอนาคต (และเพื่อนร่วมทีม) สิ่งที่ทีมมืออาชีพยึดถือ:

  • บรรทัดแรก (subject) สั้น < 50 ตัวอักษร — สรุปว่า commit นี้ทำอะไร ขึ้นต้นด้วยตัวพิมพ์ใหญ่ ไม่ต้องมีจุดท้ายประโยค
  • ใช้ imperative mood (รูปคำสั่ง) เช่น "Add", "Fix", "Remove" ไม่ใช่ "Added" หรือ "Fixes" — อ่านเป็น "ถ้า apply commit นี้แล้วจะ ____"
  • เว้นบรรทัดว่าง แล้วตามด้วย body อธิบาย why (ทำไมถึงแก้) ไม่ใช่ what (โค้ดบอก what อยู่แล้ว)
# ❌ ไม่ดี
fixed stuff
update
asdf

# ✅ ดี — subject สั้น, imperative, มี body อธิบาย why
Add rate limiting to login endpoint

Brute-force attempts spiked last week. Limit to 5 tries
per IP per minute to slow down credential stuffing.

Atomic commit: หนึ่ง commit ควรทำ เรื่องเดียวจบ ถ้าคุณแก้บั๊ก + เพิ่มฟีเจอร์ + จัดฟอร์แมต ในคราวเดียว ให้แยกเป็น 3 commit (ใช้ git add -p ช่วย) เพื่อให้ revert/review/หา bug ทีหลังง่าย ประวัติที่ดีเล่าเรื่องทีละก้าว ไม่ใช่ก้อนใหญ่ก้อนเดียว

8. chain ของ commit ที่ atomic

เมื่อแต่ละ commit ทำเรื่องเดียว ประวัติจะอ่านเป็นเรื่องราวได้ทันที — เลื่อนดู step ถัดไปได้

9. git commit --amend: แก้ commit ล่าสุด

เพิ่ง commit ไปแล้วนึกได้ว่าพิมพ์ message ผิด หรือลืม add ไฟล์? ไม่ต้องสร้าง commit ใหม่ ใช้ --amend แก้ commit ล่าสุดได้เลย

# แก้แค่ message ของ commit ล่าสุด
git commit --amend -m "Add login form validation"

# ลืม add ไฟล์ — stage เพิ่มก่อน แล้ว amend (--no-edit = ไม่แก้ message เดิม)
git add forgotten.js
git commit --amend --no-edit
🚨
amend เปลี่ยน hash! --amend ไม่ได้ "แก้" commit เดิมจริง ๆ — มันสร้าง commit ใหม่ มาแทนที่ ทำให้ได้ hash ใหม่ ดังนั้น ห้าม amend commit ที่ push ขึ้น remote ที่คนอื่นใช้แล้ว เพราะประวัติจะแตกกัน amend เฉพาะ commit ที่ยังอยู่ในเครื่องคุณคนเดียวเท่านั้น

11. Workshop

🛠️
ภารกิจ: ฝึกควบคุม staging และแก้ commit ล่าสุด ทำใน terminal จริง พร้อมดู git status และ git diff ทุกขั้น
  1. git init ในโฟลเดอร์ใหม่ สร้าง index.html และ style.css
  2. git add index.html (เฉพาะไฟล์เดียว) → git status สังเกตว่า style.css ยัง untracked
  3. git commit -m "Add homepage skeleton" เขียน subject สั้น < 50 ตัว แบบ imperative
  4. แก้ index.html 1 บรรทัด → รัน git diff (เห็นความต่าง) → git add index.html → git diff (ว่าง!) → git diff --staged (เห็นความต่างกลับมา)
  5. เผลอ commit ทั้งที่ลืม style.css: git add style.css แล้ว git commit --amend --no-edit → git log สังเกตว่า hash ของ commit ล่าสุด เปลี่ยนไป
เป้าหมาย: เห็นว่าคุณเลือกได้ว่าจะ commit อะไร และแก้ commit ล่าสุดได้โดยไม่ทิ้งขยะไว้ในประวัติ

สรุป

  • commit บันทึก snapshot ของ เฉพาะสิ่งที่อยู่ใน staging area — จึงต้อง git add ก่อนเสมอ
  • ไฟล์เดินผ่านวงจร untracked → staged → committed → modified อ่านได้จาก git status
  • git diff = working vs staging, git diff --staged = staging vs commit ล่าสุด
  • เขียน message subject สั้น < 50 ตัว, imperative, body อธิบาย why และทำ atomic commit (ใช้ git add -p ช่วยแยก)
  • git commit --amend แก้ commit ล่าสุดได้ แต่ เปลี่ยน hash — ห้ามใช้กับ commit ที่ push แชร์ไปแล้ว
