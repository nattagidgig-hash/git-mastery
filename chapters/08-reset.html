Reset

git reset คือการ "ย้อนเวลา" ด้วยการเลื่อน pointer ของ branch (และ HEAD) ไปยัง commit ที่เลือก หัวใจอยู่ที่สามโหมด — --soft, --mixed, --hard — ซึ่งต่างกันที่ว่ามัน "แตะ" tree ไหนบ้าง เข้าใจตรงนี้แล้ว reset จะไม่ใช่คำสั่งน่ากลัวอีกต่อไป

🎯 Learning Objectives
  • เข้าใจว่า reset คือการ เลื่อน branch pointer + HEAD ไปยัง commit เป้าหมาย
  • แยกแยะสามโหมด --soft / --mixed / --hard ว่ากระทบ Working & Staging อย่างไร
  • ใช้ git reset HEAD <file> เพื่อ unstage และ git reset --hard HEAD~1 เพื่อลบ commit ล่าสุด
  • รู้ความอันตรายของ --hard และรู้ว่า reflog ช่วยกู้ commit ที่ "หาย" ได้

1. reset คืออะไรจริง ๆ

ในบท Mental Model เราเห็นแล้วว่า HEAD ชี้ commit ปัจจุบัน (ผ่าน branch) และมีสาม tree เสมอ: Working Directory, Staging Area (Index), และ Repository. คำสั่ง git reset มีหน้าที่หลักเดียวคือ ย้าย branch pointer (และ HEAD) ไปยัง commit ที่เราระบุ ส่วนจะลากเอา Staging กับ Working ตามไปด้วยหรือไม่นั้น ขึ้นกับโหมด

สังเกตว่า reset ไม่ได้สร้าง commit ใหม่ (ต่างจาก revert) มันแค่เลื่อน pointer ถอยหลัง ทำให้ commit ที่อยู่ "ข้างหน้า" หลุดออกจากสายประวัติของ branch (แต่ object ยังอยู่ใน .git ชั่วคราว — ดูหัวข้อ reflog)

💡
ประโยคเดียวที่ต้องจำ: "reset = ขยับ branch ไปชี้ commit อื่น แล้วเลือกว่าจะลาก index/working ตามไปแค่ไหน" — soft ลากน้อยสุด, hard ลากทั้งหมด

2. สามโหมด แตะ tree ไหนบ้าง

นี่คือหัวใจของทั้งบท ภาพข้างล่างแสดงว่าแต่ละโหมด "ลาก" tree ไหนให้ขยับตาม HEAD ไปด้วย ยิ่งโหมดแรง ยิ่งกินพื้นที่มากขึ้นจากซ้ายมาขวา

โหมดHEAD/branchStaging (Index)Working Dirใช้ตอนไหน
--softเลื่อนคงเดิมคงเดิมรวมหลาย commit / แก้ message (งานทั้งหมดยังอยู่ใน staging พร้อม commit ใหม่)
--mixed (default)เลื่อนล้างคงเดิมunstage ไฟล์ / เอา commit ออกมาเป็นการแก้ที่ยังไม่ stage
--hardเลื่อนล้างทิ้งโยนงานที่ยังไม่ commit ทิ้งทั้งหมด ให้ตรงกับ commit เป้าหมายเป๊ะ ⚠️

3. ตัวอย่างสายประวัติ: reset ทำอะไรกับ branch

สมมติ branch main มี 4 commit ถ้าเรารัน git reset --hard HEAD~1 (หรือ git reset --hard a3) pointer ของ main จะถอยกลับ 1 ก้าว commit d4 หลุดออกจากสายประวัติทันที

กดปุ่ม prev/next บนกราฟเพื่อดูก่อน/หลัง reset — สังเกตว่า d4 หายไปจากสาย main แต่จริง ๆ object ยังอยู่ใน .git และเข้าถึงได้ผ่าน reflog ชั่วระยะหนึ่ง

4. คำสั่งจริง (เทียบสามโหมด)

เริ่มจากสถานะเดียวกัน: HEAD อยู่ที่ commit ล่าสุด แล้วลองสามโหมดกับ HEAD~1

# --soft: เลื่อน HEAD กลับ 1 ก้าว, การเปลี่ยนแปลงของ commit เดิมยังอยู่ใน staging
git reset --soft HEAD~1
git status
Changes to be committed:      # staged พร้อม commit ใหม่ทันที
  modified:   app.js

# --mixed (default): เลื่อน HEAD + ล้าง staging แต่ไฟล์ยังอยู่ในมือ
git reset HEAD~1               # ไม่ใส่ flag = --mixed
git status
Changes not staged for commit:  # ต้อง add ใหม่ถ้าจะ commit
  modified:   app.js

# --hard: เลื่อน HEAD + ล้าง staging + เขียนทับ working ⚠️ งานที่ยังไม่ commit หายหมด
git reset --hard HEAD~1
git status
nothing to commit, working tree clean
🔥
git reset --hard ลบงานที่ยังไม่ commit อย่างถาวร — ไฟล์ใน Working Directory ที่แก้ค้างไว้แต่ยังไม่ commit จะถูกเขียนทับและ กู้ไม่ได้ ผ่าน Git (เพราะมันไม่เคยถูกบันทึกเป็น object เลย) ส่วน commit ที่ถูก reset ออก ยังกู้ได้ผ่าน git reflog — เราจะเจาะลึก reflog ในบทถัดไป ๆ ของชุดนี้ จำไว้: ก่อน --hard ทุกครั้ง ถาม "งานที่ยังไม่ commit มีอะไรที่เสียดายไหม"

5. การใช้ที่พบบ่อยที่สุด: unstage ด้วย reset

สถานการณ์: เผลอ git add ไฟล์ที่ยังไม่อยากเอาเข้า commit นี้ ใช้ --mixed (default) เพื่อ ดึงออกจาก staging โดยไม่แตะเนื้อไฟล์เลย

git add .
git reset HEAD secret.env        # unstage ไฟล์เดียว (รูปแบบ git reset [HEAD] <file>)
Unstaged changes after reset:
M       secret.env            # ยังแก้อยู่ แค่ไม่ staged แล้ว

รูปแบบ git reset <file> นี้ ไม่เลื่อน HEAD (เพราะ commit เป้าหมายเป็น HEAD เดิม) มันแค่ทำให้ entry ของไฟล์นั้นใน Index กลับไปตรงกับ HEAD = unstage นั่นเอง Git รุ่นใหม่แนะนำ git restore --staged <file> ซึ่งทำงานเหมือนกันแต่ชื่อชัดเจนกว่า

✅
จำง่าย: git add = ดันเข้า staging, git reset <file> = ดึงออกจาก staging ทั้งคู่ไม่แตะเนื้อไฟล์ใน Working Directory

6. reset vs revert vs checkout (อย่าสับสน)

คำสั่งทำอะไรเลื่อน HEAD?สร้าง commit ใหม่?ปลอดภัยกับ history ที่ push แล้ว?
git resetเลื่อน branch pointer ถอยหลัง (ลบ commit ออกจากสาย)ใช่ไม่ไม่ — เขียนประวัติใหม่ อันตรายถ้า push ไปแล้ว
git revertสร้าง commit ใหม่ที่กลับด้านการเปลี่ยนแปลงไม่ (ขยับไปหน้า)ใช่ใช่ — ประวัติเดิมยังอยู่ครบ
git checkout/restoreดึงไฟล์/ย้าย HEAD ไปดู commit อื่น (ไม่ย้าย branch)เฉพาะ detachedไม่ใช่ (อ่านอย่างเดียว)
⚠️
ถ้า commit ถูก push ขึ้น remote แล้ว อย่าใช้ reset เพื่อลบมัน เพราะจะทำให้ประวัติเพื่อนร่วมทีมพัง — ใช้ git revert แทน (บทถัดไป) reset เหมาะกับ commit ที่ยังอยู่แค่ในเครื่องคุณ

8. Workshop

🛠️
ภารกิจ: เปิด terminal จริงในเครื่อง สร้าง repo ทดลอง แล้วเล่นกับสามโหมด พร้อมสังเกต git status และ git log --oneline ทุกขั้น
  1. สร้างโฟลเดอร์ใหม่ → git init → สร้างไฟล์ a.txt ใส่ข้อความ → git add a.txt → git commit -m "c1"
  2. แก้ a.txt อีกครั้ง → git commit -am "c2" ตอนนี้มี 2 commit (git log --oneline เพื่อยืนยัน)
  3. ลอง git reset --soft HEAD~1 แล้วดู git status — การแก้ของ c2 ควรอยู่ใน "Changes to be committed" (staged)
  4. git commit -m "c2 ใหม่" เพื่อกลับมาเป็น 2 commit แล้วลอง git reset HEAD~1 (mixed) — คราวนี้การแก้ควร ไม่ staged
  5. add + commit กลับเป็น 2 commit อีกครั้ง สร้างไฟล์ใหม่ scratch.txt (ยังไม่ commit) แล้วลอง git reset --hard HEAD~1 — สังเกตว่า commit หายไป 1 และ scratch.txt ที่ยังไม่ commit ก็หายด้วย
  6. รัน git reflog แล้วลองหา hash ของ commit ที่เพิ่ง reset ออก (บอกใบ้: บรรทัดบนสุด ๆ) — เตรียมพร้อมสำหรับบท Reflog
เป้าหมาย: รู้สึกได้ด้วยมือว่าโหมดไหน "ลาก" tree ไหนตามไป และเห็นว่า --hard ต่างจากสองตัวแรกตรงที่กินถึง Working Directory

สรุป

  • git reset = เลื่อน branch pointer + HEAD ไปยัง commit เป้าหมาย ไม่สร้าง commit ใหม่
  • --soft: เลื่อน HEAD อย่างเดียว (staging + working เก็บไว้) → รวม commit / แก้ message
  • --mixed (default): เลื่อน HEAD + ล้าง staging (working เก็บไว้) → ใช้ unstage
  • --hard: เลื่อน HEAD + ล้าง staging + ทิ้ง working = อันตราย ลบงานที่ยังไม่ commit
  • git reset HEAD <file> = unstage ไฟล์; git reset --hard HEAD~1 = ลบ commit ล่าสุดทิ้ง
  • commit ที่ถูก reset ออกยัง กู้ได้ผ่าน reflog แต่ working ที่ยังไม่ commit ไม่มีทางกู้ — อย่าใช้ reset กับ commit ที่ push แล้ว ให้ใช้ revert
